# レビューガイド

必ず日本語でレビューを実施。

## レビュー方針

- 問題が存在するという高い確信度（80%超）がある場合にのみコメントしてください。
- 簡潔に：可能な限り、コメントは1文にしてください。
- 観察結果ではなく、実用的なフィードバックに焦点を当ててください。
- テキストを確認する際は、テキストが本当にわかりにくい場合、またはエラーにつながる可能性がある場合にのみ、明瞭性の問題についてコメントしてください。
- 変更箇所に関わる関連箇所を深く確認し、今回の変更により不具合が生じる可能性のあるところも指摘してください。


## レビュー時のprefixについて

- レビューする際には、以下のprefix(接頭辞)をつけてください
[must]
[imo] (in my opinion)
[nits](nitpick)
[ask]
[fyi]

## 優先確認箇所

### セキュリティ・安全性

* 正当な理由のない unsafe なコードブロック
* コマンドインジェクションリスク（シェルコマンド、ユーザー入力）
* パストラバーサル脆弱性
* 認証情報の露出やハードコードされたシークレット
* 外部データの入力バリデーション欠落
* 機密情報が漏洩しうる不適切なエラーハンドリング

### 正確性の問題

* 非同期コードの競合状態
* リソースリーク（ファイル・接続・メモリ）
* 境界値やオフバイワンエラー
* 有用な情報を持たないエラーコンテキスト
* 不必要なチェックを含む過度に防衛的なコード
* 明白なコードを繰り返すだけの不要なコメント

## コメント不要な項目

以下についてはコメントしないでください：

* テストの失敗
* 依存パッケージの欠落（CI がカバー）
* 細かい命名の提案
* コメント追加の提案
* 実際のバグを修正しないリファクタリング
* 1コメントに複数の問題を含めること
* 意味に影響しないテキストの些細な正確さ
