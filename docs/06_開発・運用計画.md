## 10. フェーズ別ロードマップ

本システムは機能範囲が広いため、3 フェーズに分割して段階的にリリースする。

### 10.1 Phase 1: MVP（最小実用プロダクト）

1on1 の基本的な運用サイクルを支援する機能に絞る。

| 機能 | 対応セクション | 概要 |
|------|---------------|------|
| **モード選択（部下側）** | 2.A | トピック・期待する対応の事前入力 |
| **AI アイスブレイク生成** | 2.A | 日報・前回メモからのきっかけ質問提示 |
| **自動議事録＆要約** | 2.B | テキストベースの書き起こし・要約・アクションアイテム抽出 |
| **満足度・納得感入力** | 2.C | 部下による事後フィードバック（5 段階 + 自由記述） |
| **次回スケジュール仮設定** | 4 | AI による次回日程・テーマの提案 |

**Phase 1 で意図的に含めないもの:**

- 音声感情解析・本音度推定（技術的信頼性が未確立。Phase 2 以降でも **テキストベースの合致度解析を主軸** とし、音声感情解析は「社内検証 N=100 で精度 85 % 以上」を採用閾値とする。閾値未達の場合は導入しない）
- MQ スコアリング（十分なデータ蓄積が前提）
- AMRP（MQ スコアに依存）
- 組織分析ダッシュボード（データ蓄積が前提）

**Phase 1 の完了基準:**

- 1on1 の予約〜実施〜事後フィードバック収集の一連のフローが稼働すること
- 自動議事録の要約精度がユーザーテスト（N=30）で「十分に実用的」と評価されること
- 個人情報保護法に準拠した同意取得フローが組み込まれていること

### 10.2 Phase 2: データ活用・評価基盤

Phase 1 で蓄積されたデータを活用し、マネジメント品質の可視化に踏み込む。

| 機能 | 対応セクション | 概要 |
|------|---------------|------|
| **1on1 品質スコアリング** | 3.1 | 発話比率・キーワード・満足度に基づく品質スコア（テキストベース） |
| **本音度推定（テキストベース）** | 3.2 | 事前入力と実対話のトピック合致度から推定（音声感情解析は使用しない） |
| **MQ スコアリング** | 6 | 4 次元の加重平均による管理職適性スコア |
| **上司へのコーチングレポート** | 2.C | 傾向分析に基づく改善提案 |
| **キャリアパス同期** | 2.C | スキルマップとの連動、教育リソース自動推薦 |
| **AI 判定レビュー機能** | 12 | 異議申し立て・人間レビューのワークフロー |

**Phase 2 開始の前提条件:**

- Phase 1 の運用が 6 ヶ月以上安定稼働し、十分なデータが蓄積されていること
- 法務レビュー（第 11 章）が完了し、MQ スコアの人事利用に関する社内ガイドラインが策定されていること

### 10.3 Phase 3: 高度分析・組織最適化

AI の高度な分析機能と組織全体への展開を行う。

| 機能 | 対応セクション | 概要 |
|------|---------------|------|
| **組織分析ダッシュボード** | 2.D | 形骸化アラート、埋もれた才能の検知 |
| **AMRP（改善プログラム）** | 7 | 診断連動型のトレーニング自動生成・モニタリング |
| **不当評価アラート** | 6.4 | 成果データと定性評価の乖離検出（人間レビュー必須） |
| **期待値調整** | 3.3 | キャリア目標と現状のギャップ分析・マイルストーン提示 |

**Phase 3 開始の前提条件:**

- Phase 2 の MQ スコアの有用性が、人事担当者の評価で確認されていること
- AI 判定レビュー機能が運用され、誤判定率が許容範囲内に収まっていること

**Phase 3 リリース時の MQ スコア算出ロジック改修:**

Phase 3 の AMRP（第 7 章）は MQ スコアへのボーナス加点を含む設計のため、Phase 3 リリース時に以下の MQ スコア算出ロジック（第 6 章）の改修が必要となる。

| 改修項目 | 内容 | 影響範囲 |
|---------|------|---------|
| ボーナス加点ロジックの組み込み | AMRP のミッション達成に応じたボーナス加点（7.5 節）を月次バッチに組み込む | `mq_scores` の算出処理 |
| raw スコアと表示スコアの分離 | タイプ判定にはボーナス加点前の raw スコアを使用する仕組みの実装 | スコアリングエンジン全体 |
| `training_programs` テーブルとの連携 | AMRP 在籍中の重み固定ロジックの実装 | 月次バッチの重み取得処理 |
| `escalated` ステータスの対応 | `training_programs.status = 'escalated'` に基づくアラート発行 | トレンド分析・通知処理 |

## 13. 技術スタック・インフラ構成

### 13.1 技術選定方針

TypeScript による全レイヤー統一を最優先とする。AI によるコード生成・レビューを前提とした開発体制を想定しており、型安全性の高い単一言語で統一することで、生成精度の向上とコンテキストスイッチの排除を実現する。AWS 運用を前提とする。

### 13.2 技術スタック一覧

| レイヤー | 採用技術 | 選定理由 |
|----------|----------|----------|
| **バックエンド** | NestJS（Node.js 20 LTS+） | TypeScript ネイティブ。DI / デコレータベースの構造化フレームワークで、大規模アプリケーションに適合 |
| **フロントエンド** | Nuxt 3（Vue 3 + TypeScript） | Vue ベースで既存 axol-clients の知見を活用可能。SSR / SSG / SPA を柔軟に切替可能 |
| **ORM** | Prisma | Aurora MySQL 対応。スキーマから TypeScript 型を自動生成し、DB 設計書（第 9 章）との整合性を型レベルで担保 |
| **データベース** | Amazon Aurora MySQL 8.0+ | MySQL 互換で高可用性。リードレプリカでダッシュボード系の読み取り負荷を分散可能 |
| **キャッシュ** | Amazon ElastiCache（Valkey / Redis 互換） | BullMQ のバックエンドとしても兼用 |
| **音声ファイル保存** | Amazon S3（暗号化：SSE-S3 or SSE-KMS） | 30 日後の自動削除は S3 ライフサイクルポリシーで実現 |
| **音声書き起こし** | OpenAI Whisper API（第 1 候補）/ Amazon Transcribe（第 2 候補） | Whisper は日本語精度が高くコスト安。Amazon Transcribe は AWS 内完結でデータ主権に優れる |
| **話者分離** | Amazon Transcribe（話者分離内蔵）or pyannote.audio on ECS | Amazon Transcribe なら STT + 話者分離が 1 サービスで完結し構成がシンプル |
| **LLM 分析** | Amazon Bedrock（Claude） | AWS 内完結でデータが AWS 外に出ない |
| **非同期処理** | BullMQ + Amazon ElastiCache | 音声処理パイプライン（STT → LLM 分析 → 結果保存）をジョブチェーンで実装。`@nestjs/bullmq` で NestJS に統合 |
| **バッチ処理** | NestJS Schedule（`@nestjs/schedule`）+ Amazon ECS Task | MQ スコア月次算出等。Cron ベースのスケジューリングを NestJS 内で完結 |
| **認証** | AWS Cognito（SAML / OIDC 対応）+ Passport.js | SSO / MFA 要件を満たす。`@nestjs/passport` で NestJS に統合 |
| **バリデーション** | class-validator + class-transformer | デコレータベースのリクエストバリデーション。DTO 定義が型安全 |
| **監視** | Amazon CloudWatch + Sentry | エラートラッキングと APM を Sentry で統合管理 |
| **CI/CD** | GitHub Actions | 既存の `.github/workflows/main.yml` を拡張 |

### 13.3 音声処理パイプライン構成図

```
Zoom/Teams/Meet Webhook or スマホ Upload
    │
    ▼
API Gateway → NestJS API（録音ファイル受信）
    │
    ▼
S3（暗号化保存、ライフサイクル 30 日削除）
    │
    ▼
BullMQ → NestJS Worker①: STT 処理
    │   └─ Amazon Transcribe or Whisper API
    │      └─ 書き起こし + 話者分離結果を session_analysis に保存
    ▼
BullMQ → NestJS Worker②: LLM 分析
    │   └─ Amazon Bedrock (Claude)
    │      └─ 要約 / アクションアイテム / 発話比率を session_analysis に更新
    ▼
通知（「議事録が準備できました」）
```

### 13.4 言語選定の補足

| 候補 | 評価 |
|------|------|
| **TypeScript（NestJS + Nuxt 3）** | ✅ **採用**。フロント・バック・バッチを単一言語で統一。型安全性が AI によるコード生成・レビューの精度を向上させる |
| **Python** | 🔶 **補助利用**。pyannote.audio 等の ML 系ライブラリが必要な場合のみ、Lambda / ECS タスクとして併用 |
| **PHP（Laravel）** | ❌ 不採用。既存 axol-clients との統一性はあるが、TypeScript 統一による型安全性と AI 開発効率を優先 |
| **Go** | ❌ 不採用。パフォーマンスは優位だが、Web アプリケーション開発の生産性が低い |

**結論:** TypeScript（NestJS + Nuxt 3 + Prisma）で全レイヤーを統一する。ML 系処理が必要な場合のみ Python の Lambda/ECS タスクを併用する。

**開発体制の前提:**

AI（Claude 等）によるコード生成を主軸とし、生成されたコードは別の AI セッションでレビューする。セキュリティ要件はチェックリストベースで人間が最終確認する。TypeScript の型情報が仕様書としても機能し、AI の生成精度を最大化する。

### 13.5 対面 1on1 のデバイス要件

| 項目 | 推奨条件 |
|------|----------|
| **録音デバイス** | スマートフォン（iOS / Android）の標準マイク |
| **話者間距離** | 1m 以内 |
| **環境** | 個室推奨。オープンスペースの場合は話者分離精度が低下する可能性あり |
| **フォールバック** | 話者分離精度が不十分な場合は手動メモモードに自動切替 |

## 14. 非機能要件

### 14.1 パフォーマンス

| 項目 | 要件 |
|------|------|
| **画面応答時間** | 通常操作は 2 秒以内、ダッシュボードの初回読み込みは 5 秒以内 |
| **音声書き起こし遅延** | リアルタイム表示は発話後 3 秒以内（ストリーミング処理） |
| **AI 要約生成** | セッション終了後 5 分以内に要約を生成・保存 |
| **MQ スコア算出** | 月次バッチ処理として、全マネージャー分を 1 時間以内に完了 |

### 14.2 可用性

| 項目 | 要件 |
|------|------|
| **稼働率** | 99.5 %（月間ダウンタイム 3.6 時間以内） |
| **計画停止** | 月 1 回、深夜帯（2:00〜5:00）に最大 3 時間のメンテナンスウィンドウを設ける |
| **障害復旧目標（RTO）** | 4 時間以内 |
| **データ復旧目標（RPO）** | 1 時間以内（1 時間前の状態まで復旧可能） |

### 14.3 スケーラビリティ

| 項目 | 初期想定 | 将来想定 |
|------|----------|----------|
| **ユーザー数** | 500 名 | 10,000 名 |
| **同時セッション数** | 50 | 500 |
| **月間セッション数** | 1,000 | 20,000 |
| **データ増加量** | 10 GB/月 | 200 GB/月 |

### 14.4 セキュリティ

| 項目 | 要件 |
|------|------|
| **通信** | TLS 1.3 以上による暗号化 |
| **保存データ** | AES-256 によるデータベース暗号化（encryption at rest） |
| **認証** | SAML 2.0 / OIDC による SSO 連携。多要素認証（MFA）を推奨 |
| **認可** | RBAC（Role-Based Access Control）を NestJS Guards・Decorators・Interceptors で実装 |
| **音声データ** | 書き起こし完了後、生音声は暗号化した上で 30 日間保持し、その後完全削除 |
| **ペネトレーションテスト** | リリース前および年 1 回の外部診断を実施 |

### 14.5 データ保持ポリシー

| データ種別 | 保持期間 | 期限後の処理 |
|-----------|----------|-------------|
| **音声生データ** | 30 日 | 完全削除（復元不可） |
| **書き起こしテキスト** | 5 年 | 完全削除 |
| **AI 要約・アクションアイテム** | 5 年 | 匿名化の上アーカイブ |
| **MQ スコア（集計データ）** | 7 年 | 匿名化の上アーカイブ |
| **フィードバック回答** | 5 年 | 完全削除 |
| **同意記録** | 永久保持 | — |
| **監査ログ** | 7 年 | 匿名化の上アーカイブ |

**追加ルール:**

- 異議申し立て（`ai_judgment_reviews`）が `pending` / `under_review` の状態にある場合、関連する全データの保持期間を **解決日から 1 年間延長** する
- 保持期間の起算日は「最終更新日」ではなく **「セッション実施日」** で統一する
