> **注記**: 本設計は Amazon Aurora MySQL 8.0+ を前提とする。アクセス制御（旧仕様で想定していた PostgreSQL の Row Level Security）は、Laravel の Policy・Gate・Middleware で実装する。

### 9.4 評価・MQ（マネジメント適性）系（Phase 2）

上司の能力を可視化するためのデータ群。Phase 2 以降で実装する。

#### `mq_scores`（MQ スコア履歴）

| カラム名 | データ型 | 制約 | 説明 |
|----------|----------|------|------|
| `mq_id` | `VARCHAR(255)` | PK | スコアレコード ID（UUID） |
| `manager_id` | `VARCHAR(255)` | FK → `users.user_id`, `NOT NULL` | 対象の上司 ID |
| `recorded_date` | `DATE` | `NOT NULL` | 算出日（月次スナップショット） |
| `trust_score` | `DECIMAL(5,2)` | `NULL` 許可, `CHECK (trust_score BETWEEN 0 AND 100)` | 信頼（$T$）。最小データ要件未達の場合は NULL |
| `growth_score` | `DECIMAL(5,2)` | `NULL` 許可, `CHECK (growth_score BETWEEN 0 AND 100)` | 育成（$G$）。最小データ要件未達の場合は NULL |
| `performance_score` | `DECIMAL(5,2)` | `NULL` 許可, `CHECK (performance_score BETWEEN 0 AND 100)` | 成果（$P$）。最小データ要件未達の場合は NULL |
| `adaptability_score` | `DECIMAL(5,2)` | `NULL` 許可, `CHECK (adaptability_score BETWEEN 0 AND 100)` | 自己改善（$A$）。最小データ要件未達の場合は NULL |
| `total_mq_score` | `DECIMAL(5,2)` | `NULL` 許可, `CHECK (total_mq_score BETWEEN 0 AND 100)` | 総合 MQ スコア。全ディメンションが NOT NULL の場合のみ算出 |
| `weight_config_id` | `VARCHAR(255)` | FK → `mq_weight_configs.config_id`, `NOT NULL` | 算出時の重み設定 ID |
| `created_at` | `DATETIME` | `NOT NULL`, `DEFAULT CURRENT_TIMESTAMP` | 作成日時 |

**インデックス:**

- `idx_mq_scores_manager_id` ON `(manager_id)`
- `idx_mq_scores_recorded_date` ON `(recorded_date)`
- `UNIQUE (manager_id, recorded_date)` — 同一月に重複算出を防止

#### `mq_weight_configs`（MQ 重み付け設定）

| カラム名 | データ型 | 制約 | 説明 |
|----------|----------|------|------|
| `config_id` | `VARCHAR(255)` | PK | 設定 ID（UUID） |
| `w_trust` | `DECIMAL(3,2)` | `NOT NULL`, `CHECK (w_trust BETWEEN 0 AND 1)` | $w_1$: Trust の重み |
| `w_growth` | `DECIMAL(3,2)` | `NOT NULL`, `CHECK (w_growth BETWEEN 0 AND 1)` | $w_2$: Growth の重み |
| `w_performance` | `DECIMAL(3,2)` | `NOT NULL`, `CHECK (w_performance BETWEEN 0 AND 1)` | $w_3$: Performance の重み |
| `w_adaptability` | `DECIMAL(3,2)` | `NOT NULL`, `CHECK (w_adaptability BETWEEN 0 AND 1)` | $w_4$: Adaptability の重み |
| `is_active` | `TINYINT(1)` | `NOT NULL`, `DEFAULT 1` | 現在有効かどうか |
| `description` | `TEXT` | `NULL` 許可 | 設定変更の理由・備考 |
| `created_at` | `DATETIME` | `NOT NULL`, `DEFAULT CURRENT_TIMESTAMP` | 作成日時 |

**制約:**

- `CHECK (w_trust + w_growth + w_performance + w_adaptability = 1.00)` — 重みの合計は 1.00

**初期値（推奨）:**

| $w_1$ (Trust) | $w_2$ (Growth) | $w_3$ (Performance) | $w_4$ (Adaptability) |
|---------------|----------------|---------------------|---------------------|
| 0.30 | 0.30 | 0.20 | 0.20 |

> Trust と Growth を重視する理由：1on1 の本質は「信頼構築」と「育成」であり、短期的な成果や行動変容よりも優先度が高いと判断する。重みは運用データの蓄積後に統計的に最適化する。

#### `skill_matrix`（スキルマップ）

| カラム名 | データ型 | 制約 | 説明 |
|----------|----------|------|------|
| `skill_id` | `VARCHAR(255)` | PK | スキルレコード ID（UUID） |
| `user_id` | `VARCHAR(255)` | FK → `users.user_id`, `NOT NULL` | 対象ユーザー ID |
| `skill_name` | `VARCHAR(255)` | `NOT NULL` | スキル項目名 |
| `current_level` | `TINYINT` | `NOT NULL`, `CHECK (current_level BETWEEN 1 AND 5)` | 現在のレベル（1-5） |
| `target_level` | `TINYINT` | `NOT NULL`, `CHECK (target_level BETWEEN 1 AND 5)` | 目標レベル（1-5） |
| `created_at` | `DATETIME` | `NOT NULL`, `DEFAULT CURRENT_TIMESTAMP` | 作成日時 |
| `updated_at` | `DATETIME` | `NOT NULL`, `DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP` | 更新日時 |

**インデックス:**

- `idx_skill_matrix_user_id` ON `(user_id)`
- `UNIQUE (user_id, skill_name)` — 同一ユーザーの同一スキル重複を防止

**制約:**

- `CHECK (target_level >= current_level)` — 目標は現在以上

### 9.8 データベース設計のポイント

1. **AI 判定の「揺らぎ」を許容する設計**
   - `mq_scores` テーブルで時系列データとして蓄積する。これにより「最近、急にスコアが下がっている」という **変化の検知** が可能となる。

2. **外部システム連携の拡張性**
   - `users.external_system_id` を用意し、外部 ATS や人事システムとの ID 紐づけに対応する。

3. **プライバシー保護（Laravel Policy + Middleware）**
   - `session_analysis` の `transcript_text`（生データ）は、当該セッションの上司・部下本人、および `role_type = 'hr'` のユーザーに限定する。アクセス制御は Laravel の Policy・Gate で実装する。
   - すべてのセンシティブデータへのアクセスは `audit_logs` に記録する。

4. **同意の履歴管理**
   - `consent_records` テーブルで同意・撤回の全履歴を保持する。UPDATE ではなく INSERT で履歴を追記する設計とし、「いつ・誰が・何に同意/撤回したか」を完全に追跡可能とする。
