# 1on1 特化型マネジメント支援システム「AXOL Loop」仕様書

## 1. システムの核となる思想

1on1 を「ただの雑談」から **「教育・評価・離職防止の統合プラットフォーム」** へ昇華させる。
上司の勘に頼らず、AI が対話の質を担保し、蓄積されたデータから「組織の健康状態」をリアルタイムに診断する。

---

## 2. 機能モジュール構成

### A. 【対話前】セットアップ＆マインドセット

#### モード選択（部下側）

- **トピック選択**: 業務進捗 / キャリア / 人間関係 / プライベート
- **期待する対応**: 助言がほしい / 傾聴してほしい / 一緒に考えてほしい / 議論したい

#### AI アイスブレイク生成（上司側）

- 部下の直近の活動（日報や前回の 1on1 メモ）をもとに、最適な 3 つの「きっかけ質問」を提示する。

#### 教育進捗プッシュ

- 「現在、部下はこのスキルの習得で躓いている可能性がある。進捗を確認すること」といった形で上司にリマインドを送信する。

---

### B. 【対話中】リアルタイム AI アシスタント

#### AI ウィスパー（上司専用画面）

- 音声解析に基づき、リアルタイムに助言を表示する。
  - 例：「現在は部下が 8 割話すフェーズである。発言を遮らないこと」
  - 例：「過去の事例では、ここで『本来の理想は？』と問いかけると効果的であった」

#### 本音度・感情センシング

- 声のトーンや言葉の選び方から「本音度（納得感）」を推測する。
- 乖離が検出された場合は、上司にマイルドな確認質問を促す。

#### 自動議事録＆要約

- 決定事項とネクストアクションを自動抽出する。

---

### C. 【対話後】フィードバック・ループ

#### 満足度・納得感入力（部下側）

- 「査定には直接影響しません」という注釈付きで、対話の質に関するフィードバックを収集する。

#### 上司へのコーチングレポート

- 傾向を分析し、改善提案を行う。
  - 例：「『人間関係』の話題になると満足度が下がる傾向がある。次回は『一緒に考える』モードを意識することを推奨する」

#### キャリアパス同期

- 対話中に話された「将来の展望」をスキルマップと同期する。
- 次回の 1on1 までに必要な教育リソースを自動ピックアップする。

---

### D. 【管理者・人事向け】組織分析ダッシュボード

#### 形骸化アラート

- 実施時間が極端に短い、または「業務進捗」の話題のみで終始している部署を特定する。

#### マネージャー評価（Management Score）

- 部下の満足度向上率、教育成果、離職率から「中間管理職としての適性」を多角的にスコアリングする。

#### 「埋もれた才能」の検知

- 成果を出しているにもかかわらず、1on1 での満足度や自己評価が低い（＝正当に評価されていないと感じている）社員を自動リストアップする。

---

## 3. ロジック・アルゴリズム仕様

### 3.1 1on1 品質スコアリング

| 項目 | 内容 |
|------|------|
| **入力** | 発話比率（上司 vs 部下）、感情の起伏、キーワードの出現頻度、終了後の満足度 |
| **出力** | 100 点満点での品質スコア |
| **連動アクション** | スコアが低いマネージャーに対し、AI が自動で「1on1 改善トレーニング」を配信する |

### 3.2 本音度推定（心理的安全性の可視化）

- **ロジック**: 事前に回答した「話したいこと」と、実際の対話内容の合致度を解析する。
- **匿名トレンド化**: 個人の回答をそのまま人事に送るのではなく、部署単位の「本音度」として可視化し、組織の潜在的なリスクを特定する。

### 3.3 期待値調整（理想と現実のバランス）

- **ロジック**: 部下の高い志向性と、現在の実力・会社の教育リソースを突合する。
- **機能**: 「現在のペースでは目標到達まで○年を要する。まずはこのステップのクリアを目標とすること」といった形で、現実的なマイルストーンを AI が提示する。

---

## 4. UI/UX の設計指針

### 「邪魔しない」インターフェース

- 上司の画面は、対話中に視線を逸らさずに済むよう、大きなアイコンと最小限のテキストで情報を伝達する。

### 安心感の醸成

- 部下側の入力画面には、常に「このデータがどのように使用されるか」を明記し、心理的安全性を確保する。

### フォロー不要の自動化

- 1on1 終了直後に、AI が「次回は○月○日、このテーマでの実施を推奨する」と次回の予約およびアジェンダを仮設定する。

## 5. 導入メリット（ビジネスインパクト）

| メリット | 詳細 |
|----------|------|
| **離職率の低下** | 「放置されている」「理解されていない」という不満を、AI が未然にキャッチし対策を講じる |
| **マネジメントの標準化** | スキルの低い上司でも、AI のガイドに従うことで一定水準以上の 1on1 が実施可能となる |
| **教育の効率化** | 1on1 を通じて、部下に必要な教育がその場で特定され、学習に結びつく |
| **公平な評価** | 「成果を出しているのに評価されていない」人材を、AI がデータ（成果へのコミット度）に基づいて検出する |

## 6. マネジメント適性スコア（MQ）算出ロジック詳細仕様

### 6.1 設計思想と基本原則

MQ スコアは、管理職の「人を活かす力」を多角的に数値化する仕組みである。人事評価の唯一の根拠とすることは想定しておらず、あくまで **「マネジメント改善のための診断ツール」** として位置づける。

#### 基本原則

| # | 原則 | 説明 |
|---|------|------|
| 1 | **多角的評価** | 単一指標ではなく 4 次元の組み合わせで判定し、特定の行動だけでスコアを操作することを構造的に困難にする |
| 2 | **傾向重視** | 単発のセッションスコアではなく、3 ヶ月以上の傾向を重視する。1 回の 1on1 で極端に高い/低いスコアが出ても、MQ への影響は限定的とする |
| 3 | **相対評価の排除** | 上司同士のランキングは行わない。各上司の「成長の軌跡」を絶対評価で追跡する |
| 4 | **データ不足時の安全策** | 十分なデータが蓄積されるまではスコアを算出せず、「データ収集中」と表示する |
| 5 | **透明性** | スコアの算出根拠は本人に開示し、ブラックボックス化しない |

### 6.2 スコア構成要素（変数の定義）

マネジメントの良し悪しを、以下の 4 つの次元（ディメンション）で定義する。

$$MQ = w_1 \cdot T + w_2 \cdot G + w_3 \cdot P + w_4 \cdot A$$

| 変数 | 項目名 | 内容 | データソース | 算出頻度 |
|------|--------|------|-------------|---------|
| $T$ | Trust（信頼・安全性） | 部下が本音で話せているか、心理的安全性があるか | 1on1 の本音度解析、事後満足度アンケート | セッションごと（集計は月次） |
| $G$ | Growth（育成貢献） | 部下のスキルマップがどれだけ埋まったか（成長速度） | スキルマトリクスの更新履歴 | 月次 |
| $P$ | Performance（成果完遂） | チーム目標の達成度と、個人の自己評価の乖離 | KPI 達成率、成果物評価 | 月次 |
| $A$ | Adaptability（自己改善） | AI のフィードバックを受けて上司が行動を変えたか | 1on1 の行動改善率（発話比率の修正など） | セッションごと（集計は月次） |

#### 各ディメンションのサブ指標一覧

| ディメンション | サブ指標 | 重み（ディメンション内） | 説明 |
|--------------|---------|----------------------|------|
| **Trust** | 発話比率スコア（$T_1$） | 30 % | 上司の発話比率が黄金比率（30〜40 %）にどれだけ近いか |
| | 深掘りスコア（$T_2$） | 25 % | 抽象的な発言に対し具体性を引き出す問いかけの回数・質 |
| | 満足度スコア（$T_3$） | 25 % | 部下の事後満足度（本音補正適用後） |
| | トピック多様性スコア（$T_4$） | 20 % | 「業務進捗」以外のトピック（キャリア・人間関係等）がどれだけ扱われたか |
| **Growth** | スキル更新速度（$G_1$） | 35 % | 部下のスキルマップの更新頻度と進捗速度 |
| | 標準習得期間比（$G_2$） | 30 % | 社内平均と比較したスキル習得の速さ |
| | 自走化率（$G_3$） | 35 % | 「上司の指示」から「部下の提案」への比率変化 |
| **Performance** | KPI 達成度（$P_1$） | 40 % | チーム全体の目標達成率 |
| | 幸福度相関（$P_2$） | 35 % | 成果と部下の満足度の相関（高成果＆低満足 = 減点） |
| | 評価公正度（$P_3$） | 25 % | 客観的成果と上司の定性評価の乖離の少なさ |
| **Adaptability** | フィードバック反映率（$A_1$） | 40 % | AI のアドバイスを次回 1on1 で実行した割合 |
| | 行動変容速度（$A_2$） | 30 % | アドバイスから行動変化までの所要サイクル数 |
| | 改善持続性（$A_3$） | 30 % | 一度改善した行動が後のセッションでも維持されているか |

### 6.3 詳細算出ロジック

#### ① Trust スコア（$T$）の算出

単なる「仲良し」ではなく、対話が健全かを AI が判定する。

##### $T_1$: 発話比率スコア

上司の発話比率を `session_analysis.talk_ratio_manager` から取得し、理想範囲（30〜40 %）との乖離をスコア化する。

```
T1 の算出ロジック:

  ratio = talk_ratio_manager（0〜100 の実数）

  if 30 ≤ ratio ≤ 40:
      T1 = 100（理想範囲内）
  elif ratio < 30:
      T1 = max(0, 100 - (30 - ratio) × 2)
      ※ 部下が話しすぎ = 上司の関与不足だが、1on1 の目的（部下に話させる）を
        考慮し、上司過多よりは軽い減点とする
  elif ratio > 40:
      T1 = max(0, 100 - (ratio - 40) × 3)
      ※ 上司が話しすぎている場合はより強く減点（1on1 の趣旨に反するため）

  減点係数の根拠:
    - ratio > 40（上司過多）: 係数 3。1on1 は部下が主体的に話す場であり、
      上司の発話過多は心理的安全性を直接損なう
    - ratio < 30（上司過少）: 係数 2。上司の関与不足も問題だが、
      部下が自発的に多く話している可能性もあるため、やや軽い減点とする

  最終 T1 = 直近 3 回のセッションの加重平均
            （最新セッション: 50 %、前回: 30 %、前々回: 20 %）
```

##### $T_2$: 深掘りスコア

AI（LLM）が 1on1 のテキストを解析し、以下のパターンを検出する。

| 検出パターン | 加点 | 検出方法 |
|------------|------|---------|
| **オープンクエスチョン** | +5 点/回 | 「なぜ」「どう思う」「具体的には」等の質問パターン |
| **パラフレーズ（言い換え確認）** | +3 点/回 | 「つまり○○ということ？」のパターン |
| **具体化の促進** | +4 点/回 | 部下の「頑張ります」等に対し「具体的にはどんなアクション？」と返した場合 |
| **感情への言及** | +3 点/回 | 「それは大変だったね」「嬉しかったんだね」等の感情ラベリング |
| **遮り・話題転換** | -5 点/回 | 部下の発言途中での割り込み、唐突な話題変更 |

```
セッション単位の算出:
  session_T2 = min(50, 各パターンの合計点)
    ※ 1 セッションあたりの上限は 50 点（過剰なスコアリングを防止）

T2 の最終算出:
  T2 = 直近 3 回の session_T2 の加重平均 × 2（0〜100 にスケーリング）
       （最新セッション: 50 %、前回: 30 %、前々回: 20 %）
  T2 = min(100, T2)
```

##### $T_3$: 満足度スコア（本音補正付き）

```
raw_satisfaction = session_feedback.satisfaction_score（1〜5 の整数）
base_T3 = (raw_satisfaction - 1) / 4 × 100（0〜100 にスケーリング）

本音補正:
  if テキスト解析で「遠慮・諦め」パターンを検出:
      honesty_penalty = 検出強度 × 15（最大 -30 点）
      T3 = max(0, base_T3 + honesty_penalty)
  else:
      T3 = base_T3

※「遠慮・諦め」パターンの例:
  - 事前に「キャリアについて話したい」と入力 → 実際の対話で一切触れていない
  - 「大丈夫です」「問題ありません」が 3 回以上連続
  - 発話量が前回比 50 % 以下に激減
```

##### $T_4$: トピック多様性スコア

```
topics = 直近 6 回の one_on_one_sessions.topic_type の分布

topic_entropy = -Σ(p_i × log2(p_i))
  ※ p_i = 各トピックの出現割合
  ※ 4 トピック均等の場合 entropy = 2.0（最大）

T4 = (topic_entropy / 2.0) × 100
  ※ 全セッションが同一トピック → 0 点
  ※ 4 トピック均等に分布 → 100 点
```

##### $T$ の最終算出

```
T = T1 × 0.30 + T2 × 0.25 + T3 × 0.25 + T4 × 0.20
```

#### ② Growth スコア（$G$）の算出

「上司が優秀か」ではなく「部下を優秀にしたか」を評価する。

##### $G_1$: スキル更新速度

```
対象: 当該上司の直属部下全員の skill_matrix レコード

updated_skills = 直近 3 ヶ月間に current_level が上昇したスキルの数
total_skills = 全部下の skill_matrix レコード数

update_rate = updated_skills / total_skills

G1 の段階評価:
  update_rate ≥ 0.30 → 100 点（3 ヶ月で全スキルの 30 % 以上が成長）
  update_rate ≥ 0.20 → 80 点
  update_rate ≥ 0.10 → 60 点
  update_rate ≥ 0.05 → 40 点
  update_rate < 0.05 → 20 点
  3 ヶ月間更新なし → 0 点
```

##### $G_2$: 標準習得期間比

```
対象: 直近 6 ヶ月間にレベルアップしたスキル

各スキルについて:
  actual_days = 実際にレベルアップに要した日数
  avg_days = 社内全体での同スキル・同レベル帯の平均所要日数

  speed_ratio = avg_days / actual_days
    ※ 1.0 = 平均並み、2.0 = 平均の 2 倍の速さ

G2 = min(100, avg(speed_ratio) × 50)
  ※ avg(speed_ratio) = 全対象スキルの speed_ratio の算術平均
  ※ 平均 1.0（社内平均並み）なら G2 = 50 点、平均 2.0（2 倍速）なら G2 = 100 点
  ※ 対象スキルがない場合は前月のスコアを据え置き
```

##### $G_3$: 自走化率

AI が 1on1 のテキストを解析し、発話パターンの変化を追跡する。

```
directive_count = 上司の「○○してください」「○○すべき」等の指示的発話の数
proposal_count = 部下の「○○したいと思います」「○○はどうでしょうか」等の提案的発話の数
total = directive_count + proposal_count

特殊ケースの処理:
  if total == 0:
      G3 = 30（発言データ不足。中立値よりやや低く設定し「関与不足」を示唆する）
  elif directive_count == 0 かつ proposal_count > 0:
      G3 = 100（部下が完全自走状態）
  elif proposal_count == 0 かつ directive_count > 0:
      G3 = 0（上司の一方的な指示のみ）
  else:
      autonomy_ratio = proposal_count / total
      G3 = min(100, autonomy_ratio × 100)

トレンド補正:
  直近 3 回の G3 が連続上昇 → +10 ボーナス
  直近 3 回の G3 が連続下降 → -10 ペナルティ
```

##### $G$ の最終算出

```
G = G1 × 0.35 + G2 × 0.30 + G3 × 0.35
```

#### ③ Performance スコア（$P$）の算出

成果だけでなく、**成果の出し方の健全性** を評価する。

##### $P_1$: KPI 達成度

```
kpi_achievement = チーム全体の KPI 達成率（0〜100+ %）

P1 の算出:
  kpi_achievement ≥ 120 % → 100 点
  kpi_achievement ≥ 100 % → 80 + (kpi_achievement - 100) × 1.0
  kpi_achievement ≥ 80 %  → 60 + (kpi_achievement - 80) × 1.0
  kpi_achievement ≥ 60 %  → 30 + (kpi_achievement - 60) × 1.5
  kpi_achievement < 60 %  → max(0, kpi_achievement × 0.5)

※ KPI データは外部システム（人事評価システム等）から API 連携で取得する
※ KPI 未設定のチームは P1 = 50 点（中立値）として扱い、他のサブ指標の重みを増やす
※ 120 % で 100 点に到達する設計意図: MQ はマネジメント能力の診断ツールであり、
   KPI の超過達成度を識別することは目的外。120 % 以上の高達成チームの
   差別化が必要な場合は、別途業績評価システムで対応する
```

##### $P_2$: 幸福度相関（バーンアウト検知）

「成果は出ているが、1on1 の満足度が極端に低い」場合、**「メンバーを消耗させて成果を出している（暴君型）」** と判定し、適性スコアを大幅に減点する。

```
avg_satisfaction = 直属部下全員の session_feedback.satisfaction_score の平均（1〜5）
kpi_level = P1 のスコア区分（高: ≥80, 中: 40〜79, 低: <40）

P2 のマトリクス判定:

                  │ 満足度 高(≥4.0) │ 満足度 中(3.0〜3.9) │ 満足度 低(<3.0)
  ────────────────┼────────────────┼───────────────────┼────────────────
  KPI 高(≥80)     │ 100（理想型）    │ 70（要注意）        │ 20（暴君型）★
  KPI 中(40〜79)   │ 80（良好）      │ 60（標準）          │ 40（低迷）
  KPI 低(<40)      │ 60（育成型）    │ 40（苦戦中）        │ 10（危機的）

★ 暴君型の検出:
  P1 ≥ 80 かつ avg_satisfaction < 3.0 の場合
  → P2 = 20 に加え、critical_alert_flag = true を発行
  → 人事担当者に「バーンアウトリスク」として通知（第 12 章参照）
```

##### $P_3$: 評価公正度

```
対象: 直属部下全員

各部下について:
  objective_score = 客観的成果指標（KPI 達成率、成果物評価等）のスコア化（0〜100）
  subjective_score = 上司の定性評価（1on1 コメント、日報コメント等）の
                     AI によるセンチメント分析結果（0〜100）

  fairness_gap = |objective_score - subjective_score|

team_avg_gap = 全部下の fairness_gap の平均

P3 の算出:
  team_avg_gap ≤ 10 → 100 点（評価が極めて公正）
  team_avg_gap ≤ 20 → 80 点
  team_avg_gap ≤ 30 → 60 点
  team_avg_gap ≤ 40 → 40 点
  team_avg_gap > 40  → max(0, 100 - team_avg_gap × 2)

特定部下への偏り検出:
  if 特定の 1 名の fairness_gap が他の部下の平均の 3 倍以上:
      → 不当評価アラートの候補としてフラグを立てる（6.7 参照）
```

##### $P$ の最終算出

```
P = P1 × 0.40 + P2 × 0.35 + P3 × 0.25
```

#### ④ Adaptability スコア（$A$）の算出

AI のフィードバックを受けて上司が行動を変えたかを計測する。

##### $A_1$: フィードバック反映率

```
対象: 直近 6 ヶ月間に AI が発行したフィードバック（コーチングレポート）

feedback_items = AI が提示した改善アドバイスの総数
reflected_items = 次回以降の 1on1 で行動変化が確認されたアドバイスの数
  ※ 行動変化の確認は session_analysis のテキスト解析 + 発話比率等の定量指標で判定

reflection_rate = reflected_items / feedback_items

A1 = min(100, reflection_rate × 100)
  ※ フィードバックが未発行の場合は A1 = 50 点（中立値）
```

##### $A_2$: 行動変容速度

```
対象: 反映が確認されたフィードバック項目

各項目について:
  cycles_to_change = フィードバック提示から行動変化確認までのサイクル数

  speed_score:
    1 サイクル（次の 1on1 で即反映） → 100 点
    2 サイクル → 75 点
    3 サイクル → 50 点
    4 サイクル以上 → 25 点

A2 = 全項目の speed_score の平均
  ※ 反映項目がない場合は A2 = 0 点
```

##### $A_3$: 改善持続性

```
対象: 過去に行動変化が確認された項目

各項目について:
  改善が確認された後の 3 セッションを追跡

  sustained_sessions = 改善状態が維持されたセッション数（0〜3）

  persistence_score:
    3 セッション維持 → 100 点
    2 セッション維持 → 66 点
    1 セッション維持 → 33 点
    維持できず → 0 点

A3 = 全項目の persistence_score の平均
  ※ 追跡対象がない場合は A3 = 50 点（中立値）
```

##### $A$ の最終算出

```
A = A1 × 0.40 + A2 × 0.30 + A3 × 0.30
```

### 6.4 重み付け設定と最適化

#### 初期重み（推奨値）

$$MQ = 0.30 \cdot T + 0.30 \cdot G + 0.20 \cdot P + 0.20 \cdot A$$

| $w_1$ (Trust) | $w_2$ (Growth) | $w_3$ (Performance) | $w_4$ (Adaptability) |
|---------------|----------------|---------------------|---------------------|
| 0.30 | 0.30 | 0.20 | 0.20 |

> Trust と Growth を重視する理由：1on1 の本質は「信頼構築」と「育成」であり、短期的な成果や行動変容よりも優先度が高いと判断する。重みは運用データの蓄積後に統計的に最適化する。

#### 重み付けの管理

重みの変更は `mq_weight_configs` テーブルで管理する。変更時は旧設定を無効化（`is_active = 0`）し、新設定を INSERT する。

| 操作 | ルール |
|------|--------|
| **変更権限** | 人事部門の管理者（`role_type = 'hr'`）のみ |
| **変更頻度** | 四半期に 1 回を上限とする |
| **影響範囲** | 変更後は翌月の月次算出から新しい重みを適用する（遡及適用はしない）。**ただし AMRP 在籍中のマネージャーの卒業条件判定には、プログラム開始時点の重みを固定適用する**（重み変更による卒業難易度の変動を防止） |
| **変更記録** | `mq_weight_configs.description` に変更理由を必須記入。`audit_logs` にも記録する |
| **制約** | $w_1 + w_2 + w_3 + w_4 = 1.00$（DB の CHECK 制約で担保） |

#### 統計的最適化（Phase 3 以降）

十分なデータが蓄積された段階で、以下の手法により重みの最適化を行う。

```
目的関数:
  「MQ スコアが高い上司の部下は離職率が低く、スキル成長速度が速い」
  という仮説を検証し、離職率・成長速度との相関が最大となる重みを探索する

手法:
  1. 全マネージャーの MQ スコア（各重みパターン）と
     部下の離職率・スキル成長速度の相関係数を算出
  2. グリッドサーチ（0.05 刻み、約 1,000 パターン）で相関係数が最大となる重みの組み合わせを特定
     ※ 初期段階はグリッドサーチで十分だが、データ量の増加に伴い
       ベイズ最適化（Optuna 等）への移行も検討する
  3. 統計的有意性（p < 0.05）を確認し、有意でない場合は初期重みを維持

実施条件:
  - データ蓄積期間: 12 ヶ月以上
  - 対象マネージャー数: 50 名以上
  - 実施頻度: 年 1 回（年度初めの重み見直し時）
```

### 6.5 スコアの正規化とキャリブレーション

#### 正規化処理

各ディメンションの raw スコアは 0〜100 の範囲に収まるように設計しているが、サブ指標の組み合わせにより理論上の分布に偏りが生じうるため、以下の正規化を適用する。

```
正規化手順:

1. 各ディメンションの raw スコアを算出
2. 全マネージャー（N ≥ 30）の分布を算出
3. 外れ値の処理:
   - 上位 1 % / 下位 1 % をウィンザライズ（上限/下限でクリップ）
4. min-max 正規化:
   normalized = (raw - min) / (max - min) × 100
   ※ ただし初年度はデータ不足のため raw スコアをそのまま使用する

キャリブレーション実施条件:
  - マネージャー数 ≥ 30 名
  - データ蓄積期間 ≥ 6 ヶ月
  - 条件未達の場合は raw スコアのまま運用し、「参考値」ラベルを付与する
```

#### 部署間キャリブレーション

部署の特性（業務内容、チーム規模）によりスコアに構造的な差が生じる可能性があるため、以下の補正を検討する。

| 補正要因 | 補正方法 | 適用フェーズ |
|---------|---------|-------------|
| **チーム規模** | 部下が 1〜2 名の上司は Growth のデータ量が少ないため、信頼区間を広く設定する | Phase 2 |
| **業務特性** | 営業部門は KPI が明確で $P$ が高くなりやすい一方、管理部門は定量化が難しい。部門カテゴリ別の平均値で補正する | Phase 3 |
| **1on1 頻度** | 週次 vs 隔週 vs 月次でデータ量が異なる。セッション数による信頼度重み付けを適用する | Phase 2 |

### 6.6 スコアの信頼度と最小データ要件

#### 最小データ要件

MQ スコアの算出にはデータの蓄積が不可欠である。以下の最小要件を満たさない場合はスコアを算出しない。

| ディメンション | 最小要件 | 要件未達時の表示 |
|--------------|---------|----------------|
| **Trust** | 直近 3 ヶ月間に 3 回以上の 1on1 セッション完了（`session_analysis` レコードあり） | 「データ収集中」 |
| **Growth** | 担当部下に 1 名以上の `skill_matrix` レコードが存在 | 「スキルマップ未設定」 |
| **Performance** | KPI データが外部連携済み、または手動入力済み | 「KPI 未連携」 |
| **Adaptability** | AI フィードバックが 2 回以上発行済み | 「フィードバック蓄積中」 |
| **MQ 総合** | 4 ディメンション全てが算出可能 | 算出不可の場合は個別スコアのみ表示 |

#### 信頼度インジケーター

スコアの横に信頼度を 3 段階で表示し、利用者が判断の参考にできるようにする。

| 信頼度 | 条件 | 表示 |
|--------|------|------|
| **高** | 直近 6 ヶ月で 12 回以上のセッション + 部下 3 名以上のフィードバック | ★★★ |
| **中** | 直近 6 ヶ月で 6〜11 回のセッション or 部下 2 名のフィードバック | ★★☆ |
| **低** | 最小データ要件は満たすが、上記に届かない | ★☆☆ |

**運用ルール:** 信頼度「低」のスコアは、人事提案や AMRP のエントリー判定には使用しない。参考値としてのみ提示する。

### 6.7 時系列分析とアラート検知

#### トレンド分析

MQ スコアは月次で `mq_scores` テーブルにスナップショットとして蓄積する。蓄積されたデータから以下のトレンドを検出する。

| トレンド | 検出条件 | アクション |
|---------|---------|-----------|
| **急激な下降** | 前月比 -15 ポイント以上 | 人事に「要確認」通知。当該上司に Insight レポートを配信する |
| **緩やかな下降** | 3 ヶ月連続で -5 ポイント以上の下降 | コーチングレポートの配信頻度を上げる（月次 → 隔週） |
| **改善傾向** | 3 ヶ月連続で +5 ポイント以上の上昇 | 本人に「改善が確認されています」というポジティブフィードバックを送信する |
| **停滞** | 6 ヶ月間で ±3 ポイント以内の変動 | 新しい観点でのコーチングレポートを試験的に配信する |
| **異常値** | 前月比 ±25 ポイント以上の変動 | スコア算出ロジックのエラーの可能性があるため、人事に確認を促す。自動的に `ai_judgment_reviews` に登録する |

#### 季節性の補正

年度末（3〜4 月）や評価期間前後は 1on1 の内容が「業務進捗」に偏りやすく、Trust や Growth のスコアが構造的に低下する傾向がある。以下の補正を適用する。

```
季節性補正:
  対象月: 3 月、4 月、9 月、10 月（評価期間前後）
  補正方法: 当該月のトピック多様性スコア（T4）について、
            過去 12 ヶ月の同月平均との差分を補正係数として加算
  上限: ±10 ポイント
  適用条件: 12 ヶ月以上のデータが蓄積されていること
```

### 6.8 中間管理職「適性判定」マトリクス

MQ スコアに基づき、上司を以下の 4 タイプに分類して人事へレポートする。

#### タイプ分類の判定ロジック

```
判定フロー:

  1. T, G, P, A の 4 ディメンションスコアを取得
  2. 以下の条件に上から順にマッチングを行い、最初に合致したタイプに分類

  ハイ・リーダー:
    条件: T ≥ 70 かつ G ≥ 70 かつ P ≥ 60 かつ A ≥ 60 かつ MQ ≥ 70
    ※ 全ディメンションが一定水準以上のバランス型

  プレイング・ザ・タイラント:
    条件: P ≥ 70 かつ (T < 50 or 部下の平均満足度 < 3.0)
    ※ 成果は出すが信頼関係に問題がある

  ナイス・サポーター:
    条件: T ≥ 60 かつ P < 50
    ※ 信頼関係は良好だが成果につながっていない

  スタンダード・マネージャー:
    条件: MQ ≥ 50 かつ 上記のいずれにも該当しない
    ※ 特定の強み・弱みが突出していない中間層。最もボリュームが多いゾーン

  形骸化マネージャー:
    条件: MQ < 50
    ※ 全体的にスコアが低い
```

#### タイプ別の詳細仕様

| タイプ | MQ 目安 | 特徴的なスコアパターン | 判定結果 | 人事へのアクション提案 | AMRP 連携 |
|--------|---------|---------------------|----------|----------------------|-----------|
| **ハイ・リーダー** | 70〜100 | T↑ G↑ P↑ A↑ | 適性：極めて高い | 次世代リーダー候補。より大きな組織を任せることを推奨する | 対象外（ロールモデル事例として匿名活用） |
| **プレイング・ザ・タイラント** | 40〜70 | P↑ T↓ | 適性：危険 | 成果は出すが離職を招く。「人間力」の矯正研修または専門職への異動を推奨する | Trust 改善プログラムを優先的に適用 |
| **ナイス・サポーター** | 40〜70 | T↑ P↓ | 適性：発展途上 | 人柄は良いが成果への詰めが甘い。「目標管理・コーチング」の訓練が必要 | Growth + Performance 改善プログラム |
| **スタンダード・マネージャー** | 50〜70 | 突出なし | 適性：標準 | 基本的なマネジメントは機能している。最も低いディメンションを特定し、次のステップを提示する | 最低ディメンションに対する軽量なコーチングレポートを配信（AMRP 登録は不要） |
| **形骸化マネージャー** | 0〜50 | T↓ G↓ P↓ A↓ | 適性：低い | 1on1 が作業化している。管理職からの異動も検討すべき | 全ディメンションの底上げプログラム |

#### タイプ遷移の追跡

タイプは月次のスコア算出ごとに再判定する。タイプの遷移を `mq_scores` の時系列データから追跡し、以下を検出する。

| 遷移パターン | 意味 | アクション |
|-------------|------|-----------|
| 形骸化 → スタンダード | 全体的な底上げ（良い兆候） | 継続支援 + ポジティブフィードバック |
| スタンダード → ナイス・サポーター or タイラント | 特定ディメンションに偏りが出現 | 低いディメンションへのコーチングレポートを配信 |
| スタンダード → ハイ・リーダー | バランスの取れた成長（理想的） | ロールモデル候補としてマーク |
| ナイス・サポーター → ハイ・リーダー | Performance も改善（理想的な成長） | ロールモデル候補としてマーク |
| ハイ・リーダー → タイラント | Trust が急落（要注意） | 緊急の Insight レポートを配信 |
| いずれか → 形骸化 | 全体的な低下（深刻） | AMRP への即時エントリーを推奨 |

### 6.9 「成果を出しているのに評価されない人材」の救済ロジック

#### 検出アルゴリズム

```
対象: 全マネージャーの直属部下

各部下について:
  1. 客観的成果スコアの算出
     objective = KPI 達成率 × 0.5 + 成果物評価 × 0.3 + 同僚からの 360 度評価 × 0.2
     ※ 360 度評価が未実施の場合は KPI × 0.6 + 成果物 × 0.4

  2. 上司の定性評価スコアの算出
     subjective = AI による 1on1 テキストのセンチメント分析
       - 肯定的言及（「よくやった」「成長した」等）→ 加点
       - 否定的言及（「まだ足りない」「期待以下」等）→ 減点
       - 言及なし → 中立（ただし成果が高い場合は「無視」として減点要素）

  3. 乖離度の算出
     gap = objective - subjective

  4. 不当評価候補の判定
     if gap ≥ 30 かつ objective ≥ 70:
         → 「成果が高いにもかかわらず評価が低い」候補としてフラグ
```

#### 段階的エスカレーション

即座に人事へ通知するのではなく、段階的に対応する。

| 段階 | トリガー | アクション | 通知先 |
|------|----------|-----------|--------|
| **Level 1: 観察** | 1 回の乖離検出（gap ≥ 30） | AI が内部フラグを立てる。外部への通知は行わない | — |
| **Level 2: 確認促進** | 3 回連続で乖離が検出 | 上司に「評価の根拠を具体的に記録してください」というリマインドを表示 | 上司本人 |
| **Level 3: 人事レビュー依頼** | Level 2 のリマインド後 2 ヶ月経過しても改善なし | 人事担当者に「確認依頼」として通知。AI 判定の不確実性を明記する | 上司本人 + 人事 |
| **Level 4: 正式調査** | 人事レビューで不当評価の疑いが確認された場合 | `ai_judgment_reviews` に登録し、正式な調査プロセスを開始する | 上司 + 人事 + 上位管理者 |

#### 誤検知への対処

不当評価アラートの誤検知を最小化するための仕組みを設ける。

| 対策 | 説明 |
|------|------|
| **複数指標の組み合わせ** | 客観スコアと主観スコアの乖離だけでなく、部下本人の満足度、他の部下との比較も加味する |
| **文脈の考慮** | 1on1 のテキスト解析で否定的な表現があっても、「改善に向けた建設的なフィードバック」として使われている場合は乖離としてカウントしない |
| **異議申し立てチャネル** | 上司は不当評価アラートに対して異議を申し立てることができる（`ai_judgment_reviews` テーブル経由） |
| **誤検知率のモニタリング** | 四半期ごとに誤検知率（異議が `upheld` = AI 判定を支持、ではなく `overturned` = 覆された割合）を集計し、閾値の調整に反映する |

### 6.10 スコアリングエンジンのアーキテクチャ

#### 処理方式

| 処理 | 方式 | タイミング | 実装 |
|------|------|-----------|------|
| **セッション単位のサブ指標算出** | 非同期ジョブ | セッション完了 + `session_analysis` 更新後 | Laravel Queue（SQS） |
| **月次 MQ スコア算出** | バッチ処理 | 毎月 1 日 02:00（JST） | Laravel Schedule + ECS Task |
| **トレンド分析・アラート検知** | バッチ処理 | 月次スコア算出完了後に連続実行 | Laravel Schedule + ECS Task |
| **タイプ判定** | バッチ処理 | 月次スコア算出完了後に連続実行 | Laravel Schedule + ECS Task |

#### 月次バッチ処理フロー

```
毎月 1 日 02:00 (JST) 開始

1. 対象マネージャーの一覧を取得
   └─ users WHERE role_type = 'manager' AND is_active = 1

2. 各マネージャーについてディメンション別スコアを算出
   ├─ T: 直近 3 ヶ月の session_analysis + session_feedback を集計
   ├─ G: skill_matrix の更新履歴 + session_analysis のテキスト解析結果を集計
   ├─ P: 外部 KPI データ + session_feedback + session_analysis を集計
   └─ A: AI フィードバック発行履歴 + session_analysis の行動変化検出結果を集計

3. 最小データ要件のチェック
   └─ 要件未達のディメンションは NULL として保存

4. 重み付け設定の取得
   └─ mq_weight_configs WHERE is_active = 1

5. MQ 総合スコアの算出
   └─ 全ディメンションが算出可能な場合のみ total_mq_score を算出

6. mq_scores テーブルに INSERT
   └─ UNIQUE (manager_id, recorded_date) により重複防止

7. トレンド分析の実行
   ├─ 前月比の変化量を算出
   ├─ 3 ヶ月トレンドを判定
   └─ 異常値・急激な変動を検出

8. タイプ判定の実行
   ├─ 6.8 の判定ロジックを適用
   └─ 前月からのタイプ遷移を検出

9. アラート・通知の発行
   ├─ 急激な下降 → 人事通知
   ├─ 改善傾向 → ポジティブフィードバック
   ├─ 異常値 → ai_judgment_reviews に自動登録
   └─ バーンアウト検知 → critical_alert_flag 発行

10. バッチ完了ログの記録
    └─ audit_logs に処理結果（対象者数、所要時間、エラー件数）を記録
```

#### パフォーマンス要件

| 項目 | 要件 |
|------|------|
| 全マネージャー（500 名想定）の月次算出 | 1 時間以内 |
| 全マネージャー（10,000 名想定）の月次算出 | 4 時間以内 |
| LLM API 呼び出し（テキスト解析部分） | 並列度 10 で実行。レートリミットに抵触しないよう制御する |
| リトライ | 各ジョブは最大 3 回リトライ。3 回失敗した場合は当該マネージャーをスキップし、人事に通知する |

| タイプ判定の通知 | `audit_logs` | `mq_scores` |
| 不当評価アラート | `ai_judgment_reviews` | `session_analysis`, `session_feedback`, 外部 KPI データ |
| トレンドアラート | `audit_logs` | `mq_scores`（時系列） |
| 信頼度算出 | — （`mq_scores` 参照時にリアルタイム算出） | `one_on_one_sessions`, `session_feedback`, `skill_matrix` |

#### `mq_scores` テーブルへの保存仕様

| カラム | 保存タイミング | 備考 |
|--------|-------------|------|
| `trust_score` | 月次バッチ | NULL 許可（最小データ要件未達の場合） |
| `growth_score` | 月次バッチ | NULL 許可 |
| `performance_score` | 月次バッチ | NULL 許可 |
| `adaptability_score` | 月次バッチ | NULL 許可 |
| `total_mq_score` | 月次バッチ | 全ディメンションが NOT NULL の場合のみ算出 |
| `weight_config_id` | 月次バッチ | 算出時点で `is_active = 1` の設定 ID を記録（再現性担保） |

### 6.12 倫理的ガードレール

#### データの匿名化・集約ルール

| データ | 上司本人への開示 | 人事への開示 | 経営への開示 |
|--------|----------------|-------------|-------------|
| MQ スコア（数値） | 開示する | 開示する（数値） | レベル表記のみ（A/B/C/D） |
| 各ディメンションスコア | 開示する | 開示する | 部署平均のみ |
| タイプ判定 | 開示しない | 開示する | 部署別の分布のみ |
| サブ指標の詳細 | 開示する | 開示する | 開示しない |
| 部下の個別フィードバック | 匿名化して開示 | 匿名化して開示 | 開示しない |

#### 本音度の取扱い

- **AI 推定の本音度は「傾向」として扱う**: 上司が部下を問い詰めることを防ぐため、個別の本音度は上司には開示せず、「部署の心理的安全性の平均値」としてのみ表示する
- 本音度の推定結果を `session_analysis` に永続化する場合は、個人を特定できない集約値のみとする

#### リベンジ防止メカニズム

1on1 の不満を記入したことで部下が不利益を被らないよう、以下のメカニズムを実装する。

| メカニズム | 説明 |
|-----------|------|
| **評価変更の監視** | 部下がフィードバックで低評価を入力した後、30 日以内に当該上司が当該部下の評価関連データ（1on1 コメント、スキルマップ等）を変更した場合、AI が変更内容の妥当性を自動チェックする |
| **不当変更の検知** | 評価変更がネガティブ方向で、かつ客観的成果データとの整合性がない場合、`ai_judgment_reviews` に自動登録する |
| **部下への透明性** | 部下は自分に関する評価関連データの変更履歴を閲覧可能。ただし、変更者名は「管理者」と表示し、直接的な上司特定を防ぐ |
| **統計的監視** | 低評価フィードバック後のネガティブな評価変更の発生率を四半期ごとに集計し、統計的に有意な偏りがある部署を人事に報告する |

#### スコアの利用制限

| 利用目的 | 可否 | 備考 |
|---------|------|------|
| マネジメント改善のための自己認識 | ○ | 本来の目的 |
| コーチングレポートの配信判断 | ○ | AI が自動判断 |
| AMRP エントリーの推奨 | ○ | 最終判断は人事 |
| 人事面談の参考資料 | △ | 「参考情報」として提示。唯一の根拠としない |
| 昇進・降格の判断材料 | △ | 他の評価指標と組み合わせて使用。MQ 単独での判断は禁止 |
| 給与・賞与の算定根拠 | × | 直接的な報酬連動は禁止 |
| 解雇・配置換えの唯一の根拠 | × | 法的リスクが高く、労務問題に発展する可能性がある |

## 7. AI マネジメント・リカバリー・プログラム（AMRP）詳細仕様

### 7.1 プログラム概要と設計思想

AMRP は MQ スコアの低い管理職に対し、AI が個別最適化された改善プログラムを自動生成・配信・モニタリングする仕組みである。「研修に行かせる」のではなく、**日常の 1on1 の中で行動を変える** ことを目的とする。

#### 設計原則

| # | 原則 | 説明 |
|---|------|------|
| 1 | **行動変容ファースト** | 知識の習得ではなく、次の 1on1 での具体的行動変化を最小単位とする |
| 2 | **スモールステップ** | 1 回のミッションで求める変化は 1 つだけとし、達成可能な粒度にする |
| 3 | **非懲罰的** | プログラムへの参加は「処分」ではなく「支援」として位置づけ、表現・UI もこれに徹する |
| 4 | **データドリブン** | 改善の進捗は主観的自己申告ではなく、1on1 の音声・テキストデータから客観的に判定する |
| 5 | **段階的介入** | 軽微な課題には軽い介入から開始し、改善が見られない場合にのみ介入レベルを上げる |

### 7.2 プログラムライフサイクル

#### エントリー条件（自動登録トリガー）

以下のいずれかの条件を満たした場合、AI がプログラムへの登録を推奨する。最終的な登録判断は人事担当者が行う。

| トリガー | 条件 | 対象ディメンション |
|---------|------|-------------------|
| **MQ スコア閾値割れ** | 4 次元のいずれかが 40 点未満（100 点満点中） | 該当ディメンション |
| **急激なスコア低下** | 前月比で 15 ポイント以上のスコア下落 | 該当ディメンション |
| **部下フィードバック連続低評価** | 直近 3 回の 1on1 で部下の満足度が連続 2 以下（5 段階中） | Trust |
| **スキルマップ停滞** | 担当部下全員のスキルマップが 3 ヶ月以上更新されていない | Growth |
| **離職リスクアラート** | 担当部下に `critical_alert_flag = true` が発生 | 全ディメンションの緊急レビュー |

#### プログラム期間と構造

```
┌─────────────────────────────────────────────────────────────────┐
│                    AMRP プログラム全体像                          │
│                                                                 │
│  Month 1          Month 2          Month 3          判定         │
│  ┌──────────┐    ┌──────────┐    ┌──────────┐    ┌─────────┐  │
│  │ Cycle 1  │    │ Cycle 3  │    │ Cycle 5  │    │ 卒業    │  │
│  │ Insight  │    │ Insight  │    │ Insight  │    │ or      │  │
│  │ Learn    │    │ Learn    │    │ Learn    │    │ 継続    │  │
│  │ Mission  │    │ Mission  │    │ Mission  │    │ or      │  │
│  ├──────────┤    ├──────────┤    ├──────────┤    │ エスカ  │  │
│  │ Cycle 2  │    │ Cycle 4  │    │ Cycle 6  │    │ レーション│  │
│  │ Insight  │    │ Insight  │    │ Insight  │    └─────────┘  │
│  │ Learn    │    │ Learn    │    │ Learn    │                  │
│  │ Mission  │    │ Mission  │    │ Mission  │                  │
│  └──────────┘    └──────────┘    └──────────┘                  │
│                                                                 │
│  ※ 1 サイクル ≈ 2 週間（1on1 の実施間隔に連動）                   │
│  ※ 最大 6 サイクル（3 ヶ月）を 1 プログラムとする                  │
└─────────────────────────────────────────────────────────────────┘
```

- **1 サイクル** = 1 回の 1on1 を挟む約 2 週間。Insight → Micro-Learning → Mission の 3 ステップを実施する
- **1 プログラム** = 最大 6 サイクル（約 3 ヶ月）。途中で卒業条件を満たせば早期卒業とする
- **同時実行制限**: 1 名の上司が同時に取り組めるプログラムは **原則最大 2 ディメンション** まで。過負荷による形骸化を防止する
- **緊急時の例外**: 担当部下に `critical_alert_flag = true`（離職リスク）が発生している場合は、人事判断により 3 ディメンション以上の同時実行を許可する。ただし、優先順位を明確にし、最も緊急度の高いディメンションから着手する

#### 卒業条件

以下のすべてを満たした場合にプログラムを「完了（completed）」とする。

| # | 条件 | 判定方法 |
|---|------|----------|
| 1 | 対象ディメンションのスコアが 60 点以上に回復 | `mq_scores` テーブルの月次スナップショット |
| 2 | 直近 2 サイクル連続でミッション達成 | `training_missions.is_completed = true` が 2 件連続 |
| 3 | 部下の満足度が改善傾向にある | 下記の部下数別条件を参照 |

**条件 3 の部下数別ルール:**

| 部下数 | 満足度条件 | 根拠 |
|--------|----------|------|
| 3 名以上 | 全部下の平均満足度が 3.5 以上（5 段階） | 複数名の平均値で安定した評価が可能 |
| 2 名 | 両名の満足度がともに 3 以上 | 平均値のブレが大きいため、下限を設けて判定する |
| 1 名 | 満足度が 3 以上、**かつ** 当該部下の満足度が プログラム開始時点から 1 ポイント以上向上 | 1 名のみの場合は相性依存リスクがあるため、絶対値だけでなく **改善幅** を重視する |

#### 早期卒業ボーナス

3 サイクル以内（1.5 ヶ月以内）に卒業条件を満たした場合、MQ スコアに **+5 ポイントのボーナス加点** を付与する。改善の速さを正当に評価し、モチベーションを維持する。

### 7.3 診断連動型のトレーニング生成ロジック

AI は MQ スコアの最も低い要素を特定し、以下の 3 ステップで改善を促す。各ステップの内容は AI（Amazon Bedrock / Claude）が過去のセッションデータを分析して個別生成する。

#### ステップ 1: Insight（気づき）

自分の行動がどのように数値に影響しているかを視覚化する。

| 可視化項目 | 表示内容 | データソース |
|-----------|---------|-------------|
| **発話比率トレンド** | 直近 6 回の 1on1 における上司/部下の発話比率を折れ線グラフで表示 | `session_analysis.talk_ratio_manager` |
| **満足度推移** | 部下の事後満足度の推移（匿名化済み、部下が複数の場合は平均値） | `session_feedback.satisfaction_score` |
| **トピック分布** | 直近の 1on1 で扱ったトピックのヒートマップ（偏りの可視化） | `one_on_one_sessions.topic_type` |
| **ベンチマーク比較** | 同規模チームを持つ他マネージャーの平均値との比較（匿名化） | `mq_scores` の部署集計 |
| **ビフォー/アフター** | プログラム開始時と現在のスコア比較（2 サイクル目以降に表示） | `mq_scores` 時系列データ |

**AI による解説文の生成:**

Insight 画面には、グラフに加えて AI が生成する **1〜2 文の平易な解説** を付与する。

> 生成例：「直近 3 回の 1on1 では、あなたの発話比率が平均 68 % でした。部下が話す時間を増やすことで、信頼スコアの改善が期待できます。」

**生成ルール:**

- 批判的な表現を避け、「○○すると改善が期待できます」という提案型の文体とする
- 数値は必ず含める（曖昧な表現を排除する）
- LLM のプロンプトに「非懲罰的トーン」の制約を明示する

#### ステップ 2: Micro-Learning（学習）

5 分で読める「実践的なコツ」を提示する。

| 項目 | 仕様 |
|------|------|
| **形式** | カード形式（1 枚 = 1 コツ）。1 サイクルあたり 2〜3 枚を配信する |
| **分量** | 1 枚あたり 200〜400 字。読了時間 1〜2 分を目安とする |
| **構成** | ① 課題の説明（なぜこれが重要か）→ ② 具体的なテクニック → ③ 1on1 での使用例（会話スクリプト付き） |
| **コンテンツ生成** | AI が対象ディメンションと診断原因に基づいて自動生成する。汎用コンテンツのテンプレートライブラリ（初期 50 件）を用意し、AI が対象者の状況に合わせてカスタマイズする |
| **既読管理** | カードの既読/未読を `training_missions` テーブルで管理する |

**コンテンツのカテゴリ体系:**

| カテゴリ | 対象ディメンション | コンテンツ例 |
|---------|-------------------|-------------|
| 傾聴スキル | Trust | 「沈黙を恐れない技術」「パラフレーズの使い方」 |
| 質問技法 | Trust / Growth | 「オープンクエスチョンの 5 パターン」「仮説質問で本音を引き出す」 |
| フィードバック技法 | Growth / Performance | 「SBI 法の実践ガイド」「ポジティブ・フィードバックの比率」 |
| コーチングモデル | Growth | 「GROW モデルの 1on1 への適用」「ティーチングとコーチングの使い分け」 |
| 評価の客観化 | Performance | 「事実と意見を分離する技術」「評価バイアスのセルフチェック」 |
| 自己改善マインド | Adaptability | 「フィードバックを受け入れる心理学」「成長マインドセットの実践」 |

#### ステップ 3: Mission（実践）

次回の 1on1 で必ず実行すべき「具体的な任務」を課す。

| 項目 | 仕様 |
|------|------|
| **粒度** | 1 サイクルにつき 1 ミッション。複数の行動変化を同時に求めない |
| **表現** | 「○○してください」ではなく「次回の 1on1 で○○を試してみましょう」という提案型 |
| **達成判定** | 3 段階：AI 自動判定（`ai_auto`）、自己申告（`self_report`）、人事手動確認（`manual`） |
| **難易度調整** | サイクルが進むにつれて段階的に難易度を上げる（下表参照） |

**ミッション難易度の段階設計:**

| サイクル | 難易度 | ミッション例（Trust の場合） |
|---------|--------|---------------------------|
| 1〜2 | **Level 1: 意識** | 「次回の 1on1 で、自分の発話比率を意識してみましょう」 |
| 3〜4 | **Level 2: 実践** | 「冒頭 10 分間は相槌と質問のみで対話し、部下の発話時間を確保しましょう」 |
| 5〜6 | **Level 3: 定着** | 「セッション全体を通じて発話比率 40 % 以下を維持し、部下から 2 つ以上の提案を引き出しましょう」 |

### 7.4 スコア別・改善メニュー

#### ケース A: Trust（信頼感）が低い ─「独演会上司」への処方箋

| 項目 | 内容 |
|------|------|
| **診断原因** | 発話比率が 70 % 超、部下の感情スコアが「停滞」 |
| **Insight 可視化** | 発話比率トレンドグラフ（自身 vs 社内平均）、部下の発話量と満足度の相関散布図 |
| **Micro-Learning** | ① 「沈黙を恐れない技術」：沈黙は部下が思考している時間であり、遮らないことが信頼構築の第一歩であること ② 「質問力」の書き換え：「順調？」という Yes/No 質問を「今、一番モヤモヤしていることは何？」というオープン質問に変換するドリル |
| **Mission Lv.1** | 「次回の 1on1 で、部下が話し終わった後に 3 秒間沈黙してから応答してみましょう」 |
| **Mission Lv.2** | 「冒頭 10 分間、相槌以外の発言を禁止するモードを適用する」 |
| **Mission Lv.3** | 「セッション全体で発話比率 40 % 以下を達成し、部下から自発的なアクションアイテム提案を引き出す」 |
| **AI 自動判定基準** | `talk_ratio_manager` が 45 % 以下 かつ 部下の発話に質問形式の応答が 3 回以上含まれること |

#### ケース B: Growth（育成）が低い ─「丸投げ・放置上司」への処方箋

| 項目 | 内容 |
|------|------|
| **診断原因** | 部下のスキルマップが 3 ヶ月間更新されていない、1on1 のトピックが「業務進捗」に偏っている |
| **Insight 可視化** | 部下ごとのスキルマップ更新頻度ヒートマップ、1on1 トピック分布の偏り度 |
| **Micro-Learning** | ① GROW モデルの実装：Goal（目標）→ Reality（現状）→ Resource（資源）→ Will（意思）を順番に聞くテンプレートの提供 ② フィードバックの SBI 法：Situation（状況）、Behavior（行動）、Impact（影響）をセットで伝える練習 |
| **Mission Lv.1** | 「次回の 1on1 で、部下のキャリア目標について 1 つ質問してみましょう」 |
| **Mission Lv.2** | 「部下のスキルマップを 1 つ選び、習得のための具体的なアクションを 3 つ一緒に決定する」 |
| **Mission Lv.3** | 「前回決定したアクションの進捗を確認し、SBI 法で具体的なフィードバックを提供する」 |
| **AI 自動判定基準** | セッション中に「キャリア」「スキル」「成長」関連のキーワードが 5 回以上出現し、`skill_matrix` の `updated_at` がセッション後 7 日以内に更新されること |

#### ケース C: Performance（評価）が不当 ─「バイアス上司」への処方箋

| 項目 | 内容 |
|------|------|
| **診断原因** | 成果データと評価スコアに乖離がある |
| **Insight 可視化** | 部下ごとの「客観的成果スコア vs 上司の定性評価」比較チャート、評価の分散度（特定部下に偏りがないか） |
| **Micro-Learning** | ① アンコンシャス・バイアス（無意識の偏見）診断：AI が抽出した「評価が偏っている部下」との過去のやり取りを振り返る ② 事実ベース評価への変換：「頑張りが足りない」という主観を「目標達成率○ %、エラー率○ %」という客観的事実に変換するトレーニング |
| **Mission Lv.1** | 「次回の 1on1 で、部下の成果を 1 つ具体的な数字で言及してみましょう」 |
| **Mission Lv.2** | 「AI が特定した『評価のズレ』について、部下に『自分の評価と本人の実感に差がある理由は何だと思うか』と直接確認する」 |
| **Mission Lv.3** | 「担当部下全員の評価根拠を事実ベース（数値・成果物・具体的行動）で記録し、AI の整合性チェックをパスする」 |
| **AI 自動判定基準** | 1on1 のテキスト中に具体的な数値・成果物への言及が 3 回以上あり、`session_analysis` の評価関連キーワードが主観表現から客観表現に変化していること |

#### ケース D: Adaptability（自己改善力）が低い ─「頑固上司」への処方箋

| 項目 | 内容 |
|------|------|
| **診断原因** | AI のフィードバックに対する行動変容が観測されない。過去 3 サイクル以上のミッション達成率が 30 % 未満 |
| **Insight 可視化** | 過去のミッション達成率の推移、AI フィードバックの受容回数 vs 無視回数、行動変容の前後比較（変化していないことの可視化） |
| **Micro-Learning** | ① 成長マインドセットの科学：「能力は固定ではなく伸ばせる」というキャロル・ドゥエックの研究をベースに、管理職向けにカスタマイズした解説 ② 「フィードバックを受け取る技術」：防衛反応を認識し、建設的に受け止めるための 3 ステップ（聞く → 確認する → 試す） ③ ロールモデル事例：AI のフィードバックを活用して MQ スコアを改善した匿名事例の紹介 |
| **Mission Lv.1** | 「AI が提示した前回のフィードバックを読み返し、『1 つだけ試せるとしたら何か』を選択してください」 |
| **Mission Lv.2** | 「選択した改善ポイントを次回の 1on1 で意識的に実践し、実践後に自己評価コメントを記入してください」 |
| **Mission Lv.3** | 「部下に『最近、自分の 1on1 の進め方で何か変化を感じたか』と直接フィードバックを求め、その内容を記録してください」 |
| **AI 自動判定基準** | ミッション選択の操作ログが記録されていること、かつ次回 1on1 の音声/テキスト解析で該当行動の兆候が検出されること |

**特記事項:** Adaptability が低いケースは、本人に改善意欲がない可能性があるため、最もデリケートな対応が求められる。Mission Lv.1 のハードルを極端に低く設定し、「まず 1 つだけ」という心理的負荷の軽減を最優先とする。

**Adaptability 専用の加速エスカレーション:** 自己変革が最も困難なディメンションであるため、通常の 7.6 のタイムラインよりも早期に外部介入を導入する。

| 経過 | トリガー | アクション |
|------|---------|-----------|
| Cycle 2 未達（約 4 週目） | Lv.1 が 2 サイクル連続未達 | 人事に「早期注意」として通知。外部コーチまたは人事面談の併用を推奨する |
| Cycle 3 未達（約 6 週目） | Lv.1 が 3 サイクル連続未達 | 7.6 の Level 3（人事通知）に移行。プログラム単独での改善は困難と判断し、人事主導の介入を開始する |

### 7.5 「行動変容」を確認するモニタリング仕様

AI はトレーニングを配信するだけでなく、実行状況を多角的に検証する。

#### 判定方法の詳細

| 判定方法 | 概要 | 適用場面 | 信頼度 |
|---------|------|---------|--------|
| **AI 自動判定（`ai_auto`）** | 音声・テキストデータから行動変化を自動検出 | 発話比率の変化、キーワード出現頻度の変化、質問形式の増減 | 中〜高 |
| **自己申告（`self_report`）** | 上司が「ミッションを実施した」と自己報告 | AI で判定困難な行動（例：メンタルモデルの変化） | 低（参考値） |
| **人事手動確認（`manual`）** | 人事担当者がセッション記録を確認して判定 | Adaptability の改善、評価バイアスの是正 | 高 |

#### AI 自動判定の具体的ロジック

```
入力:
  - 前回セッションの session_analysis データ（ベースライン）
  - 今回セッションの session_analysis データ（比較対象）
  - 当該サイクルのミッション内容（training_missions.title）

判定プロセス:
  1. ミッションの種類に応じた評価指標を選択
     - 発話比率系 → talk_ratio_manager の変化量
     - 質問力系 → テキスト中のオープンクエスチョン出現数
     - スキル育成系 → skill_matrix の更新有無
     - 評価客観化系 → テキスト中の数値・事実表現の比率

  2. ベースラインとの比較で改善度を算出
     - 改善度 = (今回の指標値 - ベースライン) / ベースライン × 100

  3. ベースライン水準に応じた動的閾値
     改善度の閾値はベースラインの水準により調整する。
     ベースラインが悪い（目標から遠い）ほど、小さな改善でも「達成」と判定する。

     - ベースラインが目標範囲から大きく乖離（乖離率 > 50 %）:
         達成: 改善度 ≥ 5 % かつ 改善方向が目標に向かっている
         部分達成: 改善度 ≥ 3 %
         未達成: 改善度 < 3 %

     - ベースラインが目標範囲から中程度に乖離（乖離率 20〜50 %）:
         達成: 改善度 ≥ 10 % かつ 絶対値が目標範囲に接近
         部分達成: 改善度 ≥ 5 %
         未達成: 改善度 < 5 %

     - ベースラインが目標範囲に近い（乖離率 < 20 %）:
         達成: 絶対値が目標範囲内に到達
         部分達成: 改善度 ≥ 5 % だが目標範囲外
         未達成: 改善が見られない

     ※ 例: 発話比率 70 %（目標 30〜40 %）の上司が 63 % にした場合、
       乖離率 > 50 % なので改善度 10 % で「達成」。
       発話比率 45 % の上司が 40.5 % にした場合は、
       乖離率 < 20 % なので目標範囲内到達により「達成」。

出力:
  - training_missions.is_completed の更新
  - training_missions.ai_feedback への判定理由コメント生成
```

#### 部下へのサイレント・アンケート

- 1on1 直後に「上司の聞き方に変化はあったか」と部下にのみ通知する
- 質問は 3 問以内、所要時間 1 分以内に収める
- 回答は匿名化して AI の判定精度向上に使用する（上司への直接開示はしない）

| # | 質問 | 回答形式 |
|---|------|---------|
| 1 | 「今回の 1on1 で、上司は十分にあなたの話を聞いてくれましたか？」 | 5 段階（1: 全くそう思わない 〜 5: 強くそう思う） |
| 2 | 「前回と比較して、上司の対話の進め方に変化を感じましたか？」 | 3 択（良くなった / 変わらない / 悪くなった） |
| 3 | 「自由コメント（任意）」 | テキスト入力（200 字以内） |

#### 改善ボーナス（MQ アップ）

行動改善が確認された場合、MQ スコアにボーナス加点を行い、上司の改善モチベーションを維持する。

| 条件 | ボーナス加点 |
|------|------------|
| ミッション達成（1 サイクル） | +2 ポイント |
| 2 サイクル連続達成 | +3 ポイント（累計 +5） |
| 早期卒業（3 サイクル以内） | +5 ポイント |
| 部下の満足度が 1 ポイント以上向上 | +3 ポイント |

**上限:** ボーナス加点の合計は 1 プログラムあたり最大 **+5 ポイント** とする。過剰なインフレを防止する。

**タイプ判定への影響制限:** ボーナス加点は MQ スコアの **表示値** にのみ反映し、タイプ判定（6.8 節）には **ボーナス加点前の raw スコア** を使用する。これにより、ボーナス加点によるタイプ遷移の操作を防止する。

### 7.6 段階的エスカレーションと最終勧告

改善が見られない場合の対応を、時系列に沿って段階的に定義する。

#### エスカレーション・タイムライン

| 時期 | 段階 | トリガー | アクション | 通知先 |
|------|------|---------|-----------|--------|
| **1〜2 週目** | Level 0: 開始 | プログラム登録 | Insight + 初回 Micro-Learning を配信 | 上司本人のみ |
| **3〜4 週目** | Level 1: 観察 | Cycle 1 のミッション結果 | 達成 → 次のサイクルへ。未達 → AI が励ましメッセージ + 難易度を下方調整 | 上司本人のみ |
| **5〜6 週目** | Level 2: 注意喚起 | Cycle 2〜3 連続未達 | AI が「改善のコツ」を追加配信。学習コンテンツの形式を変更（テキスト → 動画 or インタラクティブ） | 上司本人のみ |
| **7〜8 週目** | Level 3: 人事通知 | Cycle 4 未達（通算 3 回以上未達） | 人事担当者に「プログラム進捗に懸念あり」として通知。人事面談の実施を推奨する | 上司本人 + 人事 |
| **9〜12 週目** | Level 4: 最終勧告 | Cycle 5〜6 未達（プログラム全体のミッション達成率 30 % 未満） | 最終レポートを発出し、配置換え等の人事判断を推奨する | 上司本人 + 人事 + 上位管理者 |

#### 最終レポートの構成

3 ヶ月間のプログラムを経ても MQ スコアが改善しない場合、システムは以下の構成で最終レポートを生成する。

| セクション | 内容 |
|-----------|------|
| **① サマリー** | 対象マネージャー名、プログラム期間、対象ディメンション、最終 MQ スコア |
| **② プログラム実施結果** | サイクルごとのミッション達成/未達の一覧、AI フィードバックへの反応履歴 |
| **③ 部下への影響分析** | 担当部下の満足度推移、離職リスクスコアの変化、スキルマップの停滞状況 |
| **④ AI の推奨アクション** | 「継続支援」「専門研修への移行」「マネジメント以外の適性についても検討の余地があります」のいずれかを、データに基づいて提示する。**「配置換え」「降格」等の直接的な人事処分を示唆する表現は使用しない**（法的リスク軽減のため） |
| **⑤ 注意書き** | 「本レポートは AI の分析に基づく参考情報であり、人事上の最終判断は人間が行うこと」を明記する |

> **AI からの最終レポート例:**
>
> 「マネージャー A 氏は、3 ヶ月間の支援プログラムにおいて改善ミッションの達成率が 17 %（6 回中 1 回達成）でした。特にフィードバック受容性（$A$ スコア）の改善が確認されていません。担当部下の満足度が低下傾向にあり、追加的な支援策の検討が必要です。マネジメント以外の領域での活躍可能性についても、本人との対話を通じて検討する余地があります。」
>
> ※ **法務上の注意:** AI の出力に「配置換え」「降格」等の具体的な人事処分を示唆する表現を含めない。裁判や労務紛争で証拠として提出された場合のリスクを考慮し、あくまで「検討の材料」として中立的な表現にとどめる。この表現方針は法務部門のレビューを経て確定すること。

### 7.7 効果測定・KPI 設計

#### AMRP 自体の成功指標

| KPI | 定義 | 目標値 | 測定頻度 |
|-----|------|--------|---------|
| **プログラム卒業率** | 卒業条件を満たして完了した割合 | 60 % 以上 | 四半期 |
| **早期卒業率** | 3 サイクル以内に卒業した割合 | 20 % 以上 | 四半期 |
| **MQ スコア改善幅** | プログラム前後の対象ディメンションスコア差の平均 | +15 ポイント以上 | 月次 |
| **部下満足度改善率** | プログラム対象上司の部下の満足度が向上した割合 | 70 % 以上 | 四半期 |
| **離職率への影響** | プログラム対象上司の部下の離職率（対象外との比較） | 対象外と同等以下 | 半期 |
| **再発率** | 卒業後 6 ヶ月以内に再度エントリー条件を満たした割合 | 20 % 以下 | 半期 |
| **ミッション達成率** | 全サイクルを通じたミッション達成の割合 | 50 % 以上 | 月次 |

#### ROI の測定方法

AMRP の投資対効果を以下の式で算出し、四半期ごとにレポートする。

| 項目 | 算出方法 |
|------|---------|
| **コスト** | AI 分析の API 利用料 + コンテンツ生成コスト + 人事担当者のレビュー工数 |
| **効果（定量）** | 離職防止による採用コスト削減額（1 名あたりの採用コスト × 離職回避人数）+ マネジメント研修費の削減額 |
| **効果（定性）** | 部下の満足度向上、組織の心理的安全性スコアの改善 |
| **ROI** | (定量効果 - コスト) / コスト × 100 % |

**ROI 算出の前提数値（初期想定）:**

| 項目 | 想定値 | 根拠 |
|------|--------|------|
| 離職 1 名あたりの採用・育成コスト | 年収の 50 %（約 300 万円） | エン・ジャパン「人材採用・育成コスト調査」等の業界水準 |
| AMRP 対象マネージャー数（初年度） | 50 名（マネージャー 500 名の 10 %） | MQ スコア閾値割れの想定比率 |
| AMRP による離職回避人数（初年度） | 5 名（対象部下の 10 %） | 控えめな推計 |
| AMRP の運用コスト（月額） | 約 50,000 円（AI API 利用料 + 人事レビュー工数） | 8.7 節のコスト試算に準拠 |

**初年度の ROI 試算:** 離職回避効果 300 万円 × 5 名 = 1,500 万円、コスト 50,000 円 × 12 ヶ月 = 60 万円。ROI ≈ 2,400 %。ただし離職回避の因果関係の立証が困難なため、**初年度は ROI の測定体制の構築を主目標** とし、2 年目以降にデータに基づく ROI の本格算出を行う。

#### DB テーブルとの連携

AMRP のデータは以下のテーブルに格納・参照される。

| AMRP の機能 | 書き込みテーブル | 参照テーブル |
|------------|----------------|-------------|
| プログラム登録・管理 | `training_programs` | `mq_scores`, `session_feedback` |
| ミッション配信・達成管理 | `training_missions` | `session_analysis`, `skill_matrix` |

| KPI 集計 | — （BI ツール / ダッシュボード向け） | `training_programs`, `training_missions`, `mq_scores`, `session_feedback` |

### 7.8 上司のプライドを傷つけない UI/UX の工夫

MQ が低い上司はプライドが高い傾向があるため、**「ダメ出し」ではなく「攻略ガイド」** として提示することが重要である。

#### 用語・表現のガイドライン

| 使用しない表現 | 代わりに使用する表現 |
|--------------|-------------------|
| 「あなたのスコアが低い」 | 「このスキルを伸ばすとチーム力がさらに向上します」 |
| 「改善が必要です」 | 「次のステップに進むための提案があります」 |
| 「ミッション未達成」 | 「次回、もう一度チャレンジしてみましょう」 |
| 「問題のある行動」 | 「より効果的な別のアプローチ」 |
| 「部下からの不満」 | 「チームからのフィードバック」 |

#### ゲーミフィケーション設計

- **レベルシステム**: 「マネジメント・レベルアップまであと 100 ポイント！」のように進捗を表示する
- **バッジ獲得**: ミッション達成ごとにバッジを付与する（「傾聴マスター」「育成の達人」等）
- **ストリーク表示**: 連続達成回数を表示し、継続のモチベーションを支える
- **ON/OFF 切替**: 年代や好みによって反感を招く可能性があるため、**個人設定でゲーミフィケーション表示の ON/OFF を切り替え可能** とする（デフォルト：ON）

#### 「上司の孤独」への寄り添い

- 「管理職は困難な立場である。このステップをクリアすればチームの成果がより効率的に出るようになる」といった、AI ならではの励ましメッセージを添える
- プログラム開始時に「この支援プログラムは、あなたのマネジメントをサポートするためのものです。評価や処分とは一切関係ありません」というメッセージを表示する
- 匿名化された「同じプログラムを経験した管理職の成功事例」を定期的に配信し、孤立感を軽減する

## 8. 音声データ処理アーキテクチャ

### 8.1 方式選定

音声データの取得・書き起こし・分析について、以下の 5 案を比較検討した。

| # | 案 | 概要 |
|---|---|------|
| 1 | Web 会議プラットフォーム API 連携 | Zoom / Meet / Teams の録音・文字起こし API を利用 |
| 2 | 自社会議ツール開発 | 専用の Web 会議 or 録音アプリを自社開発 |
| 3 | 専業ベンダー連携 | Zenkigen (harutaka) / RevComm (MiiTel) / ACES Meet 等と連携 |
| 4 | STT API + LLM 組み合わせ | 書き起こしは Whisper API / Google Cloud STT、分析は LLM で自社構築 |
| 5 | Meeting Bot 方式 | Zoom / Teams / Meet に Bot を参加者として投入し録音・書き起こし |

#### 比較評価

| 評価軸 | 案 1: API 連携 | 案 2: 自社ツール | 案 3: ベンダー | 案 4: STT + LLM | 案 5: Bot |
|--------|--------------|----------------|--------------|----------------|-----------|
| **開発コスト** | 中 | 極大 | 小〜中 | **小〜中** | 中 |
| **日本語精度** | 各社依存 | 自社次第 | 高い | **高い** | STT 依存 |
| **対面 1on1 対応** | 不可 | 可能 | 可能 | **可能** | 不可 |
| **データ主権** | 各社経由 | 完全自社 | ベンダー依存 | **STT 経由、分析は自社** | 自社管理可能 |
| **ユーザーの行動変更** | 不要 | 大 | 中 | **不要〜小** | 小 |
| **リアルタイム処理** | 限定的 | 可能 | ベンダー依存 | **ストリーミング STT で可能** | 可能 |
| **ベンダーロックイン** | 高 | なし | 高 | **低** | 低〜中 |

#### 選定結果

- **案 2（自社ツール）: 不採用** — 音声通話基盤の開発は本システムの本質（1on1 の質の向上）から外れた過大な投資となる
- **案 3（専業ベンダー）: Phase 3 以降の選択肢として温存** — 音声感情解析に強みがあるが、MVP では感情解析を含めない方針のため現時点では不要
- **案 5（Meeting Bot）: 補助手段として検討** — 1on1 の場に「第三者（Bot）」の存在を意識させる心理的抵抗がある。案 4 の録音取得手段の 1 つとしては有効

**採用方針: 案 1 + 案 4 のハイブリッド**

### 8.3 採用理由

1. **ユーザーの行動変更が最小**: 既存の Zoom / Teams / Meet をそのまま使い続けられる
2. **対面 1on1 もカバー可能**: スマホ録音 → アップロードで対応し、Web 会議に限定しない
3. **日本語書き起こし精度が十分**: Whisper API は日本語の精度が高く、コストも低い
4. **分析の自由度が高い**: LLM による分析はプロンプト・ロジックを自社で制御できるため、AXOL Loop の要件に合わせた柔軟なカスタマイズが可能
5. **STT の差替が容易**: インターフェースを統一することで、Whisper → Google Cloud STT → 国産 STT など、状況に応じた切替が可能

### 8.4 各プラットフォームの API 対応方針

| プラットフォーム | 録音取得方法 | 書き起こし | 備考 |
|-----------------|-------------|-----------|------|
| **Zoom** | Zoom Cloud Recording API（録音完了時の Webhook → ダウンロード URL 取得） | 自社 STT（Whisper API） | Zoom 標準の文字起こしは精度が不十分なため自社 STT を使用 |
| **Microsoft Teams** | Microsoft Graph API（`/communications/callRecords`） | 自社 STT（Whisper API） | Teams 標準 Transcription も日本語精度は高いが、統一性のため自社 STT を優先 |
| **Google Meet** | Google Workspace Events API（録音ファイルを Google Drive から取得） | 自社 STT（Whisper API） | Meet の録音機能は Business Standard 以上のプランが必要 |
| **対面 1on1** | 専用モバイルアプリまたは Web UI からの手動アップロード | 自社 STT（Whisper API） | 録音開始前に双方の同意確認 UI を表示する |

### 8.5 音声データの処理フロー

```
1. セッション終了（または録音ファイルアップロード完了）
   │
2. Webhook / ポーリングで録音ファイルの準備完了を検知
   │
3. 録音ファイルをダウンロードし、一時ストレージ（S3 等）に暗号化保存
   │
4. STT API に送信し、書き起こしテキストを取得
   │  ├─ 話者分離結果（上司 / 部下のラベル付け）
   │  └─ タイムスタンプ付きテキスト
   │
5. 書き起こしテキストを session_analysis テーブルに保存
   │
6. LLM API に送信し、以下を生成
   │  ├─ AI 要約（ai_summary）
   │  ├─ アクションアイテム（action_items）
   │  └─ 発話比率の算出（talk_ratio_manager）
   │
7. 分析結果を session_analysis テーブルに更新
   │
8. 音声生データの削除スケジュールを登録（30 日後に自動削除）
   │
9. セッション参加者に通知（「議事録が準備できました」）
```

### 8.6 話者分離（Speaker Diarization）の方式

1on1 は参加者が 2 名のため、話者分離の難易度は比較的低い。

| 方式 | 概要 | 採用フェーズ |
|------|------|-------------|
| **Whisper + pyannote.audio** | Whisper で書き起こし後、pyannote.audio で話者分離を行い、結果をマージする | Phase 1 |
| **Google Cloud STT v2 話者分離** | STT と話者分離を一括で処理（`enable_speaker_diarization` オプション） | Phase 1（代替案） |
| **声紋登録方式** | 事前にユーザーの声紋を登録し、高精度な話者識別を行う | Phase 2 以降（精度向上が必要な場合） |

**Phase 1 での推奨**: Whisper + pyannote.audio の組み合わせを第 1 候補とする。2 名の会話における分離精度が十分でない場合は、Google Cloud STT v2 に切り替える。

### 8.7 コスト試算（Phase 1 運用時）

前提: 月間 1,000 セッション、平均 30 分/セッション

| 項目 | 単価 | 月間コスト（税別） |
|------|------|-------------------|
| **Whisper API** | $0.006/分 | 1,000 × 30 分 × $0.006 = **$180（約 27,000 円）** |
| **LLM API（要約 + アクションアイテム抽出）** | 約 $0.02/セッション（入力 3,000 トークン + 出力 500 トークン想定） | 1,000 × $0.02 = **$20（約 3,000 円）** |
| **音声一時ストレージ（S3）** | $0.025/GB（30 分 ≈ 15 MB） | 1,000 × 15 MB × $0.025/GB = **$0.4（約 60 円）** |
| **合計** | | **約 30,000 円/月** |

> スケーラビリティ目標の 20,000 セッション/月（第 13 章）でも月額約 600,000 円であり、専業ベンダーのライセンス費用（数百万円/月〜）と比較して大幅に低コストである。

**初期値（推奨）:**

| $w_1$ (Trust) | $w_2$ (Growth) | $w_3$ (Performance) | $w_4$ (Adaptability) |
|---------------|----------------|---------------------|---------------------|
| 0.30 | 0.30 | 0.20 | 0.20 |

> Trust と Growth を重視する理由：1on1 の本質は「信頼構築」と「育成」であり、短期的な成果や行動変容よりも優先度が高いと判断する。重みは運用データの蓄積後に統計的に最適化する。

### 9.8 データベース設計のポイント

1. **AI 判定の「揺らぎ」を許容する設計**
   - `mq_scores` テーブルで時系列データとして蓄積する。これにより「最近、急にスコアが下がっている」という **変化の検知** が可能となる。

2. **外部システム連携の拡張性**
   - `users.external_system_id` を用意し、外部 ATS や人事システムとの ID 紐づけに対応する。

3. **プライバシー保護（Laravel Policy + Middleware）**
   - `session_analysis` の `transcript_text`（生データ）は、当該セッションの上司・部下本人、および `role_type = 'hr'` のユーザーに限定する。アクセス制御は Laravel の Policy・Gate で実装する。
   - すべてのセンシティブデータへのアクセスは `audit_logs` に記録する。

4. **同意の履歴管理**
   - `consent_records` テーブルで同意・撤回の全履歴を保持する。UPDATE ではなく INSERT で履歴を追記する設計とし、「いつ・誰が・何に同意/撤回したか」を完全に追跡可能とする。

## 11. 法務・コンプライアンス要件

### 11.1 個人情報保護法への対応

本システムは従業員の音声データ・対話内容・心理的傾向を扱うため、個人情報保護法（令和 5 年改正対応）における「要配慮個人情報」に該当する可能性がある。

#### 利用目的の特定と通知

| 項目 | 対応方針 |
|------|----------|
| **利用目的の特定** | 「1on1 の品質向上」「マネジメントスキルの改善支援」を利用目的として明示する。「人事評価への直接利用」を目的に含めるかは、社内規定と労使協議で決定する |
| **本人への通知方法** | システム初回利用時に利用目的・データの取扱い方針をポップアップで表示し、明示的な同意（オプトイン）を取得する |
| **第三者提供の制限** | 人事部門への共有は「第三者提供」に該当しうるため、事前同意または社内利用の正当な範囲であることを明確にする |

#### データ最小化の原則

- 音声の生データは書き起こし完了後、速やかに削除する（保持期間は最大 30 日）
- 書き起こしテキストの保持期間は 5 年間とし、期限到来後は自動削除する
- MQ スコア等の集計データは匿名化の上、7 年間保持する

### 11.2 録音・音声解析に関する同意設計

#### 同意取得フロー

```
1. システム利用開始時
   └─ 利用規約・プライバシーポリシーへの包括同意（必須）

2. 各 1on1 セッション開始時
   ├─ 上司：「本セッションを録音・AI 解析しますか？」 → Yes / No
   └─ 部下：「本セッションを録音・AI 解析しますか？」 → Yes / No
       └─ 双方が Yes の場合のみ録音を開始する

3. 録音なしの場合のフォールバック
   └─ 手動メモ入力モードに切り替え、テキストベースの要約・アクションアイテム抽出のみ実施する
```

#### 同意撤回への対応

- 同意はいつでも撤回可能とする
- 撤回後は、当該ユーザーに関する未処理の音声データを 72 時間以内に削除する
- 撤回によって不利益が生じない旨を、UI 上に明記する

### 11.3 オプトアウト設計

| レベル | 範囲 | 影響 |
|--------|------|------|
| **録音オプトアウト** | 音声録音・音声解析を拒否 | 手動メモモードで 1on1 は継続可能。自動議事録の精度は低下する |
| **AI 解析オプトアウト** | AI による分析全般を拒否 | 議事録は手動入力、フィードバック収集はアンケート形式のみ。MQ スコアの算出対象から除外する |
| **システム利用オプトアウト** | AXOL Loop の利用自体を拒否 | 従来どおりの 1on1 運用に戻る。ただし、所属部署のデータ傾向からは除外される |

**重要**: オプトアウトした従業員に対する人事上の不利益は厳禁とし、就業規則に明記する。

### 11.4 `personality_type` の取扱い

旧仕様で定義していた `personality_type`（AI 判定の性格タイプ）は、以下のリスクから **DB への永続化を廃止** する。

- **差別的運用のリスク**: 性格分類が採用・昇進判断に不適切に利用される可能性
- **ラベリングの固定化**: AI の一時点の判定が永続的なレッテルとなる危険性
- **法的リスク**: 要配慮個人情報に該当する可能性

代替として、AI がアイスブレイク質問を生成する際に一時的にコミュニケーション傾向を推定し、セッション終了後は破棄する設計とする。

## 12. AI 判定レビュー・エラーハンドリング設計

### 12.1 基本方針

**AI の判定は「参考情報」であり「最終決定」ではない。** 人事上の意思決定に影響する判定（MQ スコア、不当評価アラート、離職リスクアラート等）には、必ず人間によるレビューを介在させる。

### 12.2 AI 判定の信頼度分類

| 信頼度レベル | 判定内容の例 | 人間レビューの要否 |
|-------------|-------------|-------------------|
| **高（自動適用可）** | 議事録の要約、アクションアイテム抽出、次回スケジュール提案 | 不要（ユーザーが事後確認・修正可能） |
| **中（通知＋確認）** | 発話比率の算出、品質スコアの提示、コーチングレポート | 上司本人が内容を確認し、不正確な場合は修正を申請可能 |
| **低（人間レビュー必須）** | 不当評価アラート、離職リスクアラート、MQ スコアに基づく人事提案 | 人事担当者によるレビューを経なければ、関係者への通知・アクションは行わない |

### 12.3 異議申し立てフロー

```
1. AI 判定の通知を受けた関係者（上司 or 部下）が「異議あり」を選択
   │
2. 異議理由の入力（自由記述、必須）
   │
3. ai_judgment_reviews テーブルに status = 'pending' で登録
   │
4. 人事レビュー担当者に通知（SLA: 5 営業日以内に着手）
   │
5. レビュー担当者が調査・判断
   ├─ upheld（AI 判定を支持）: 判定を維持し、申立者に理由を通知
   ├─ overturned（AI 判定を覆す）: 判定を無効化し、関連スコアを再計算
   └─ dismissed（申立てを却下）: 根拠不十分として却下し、理由を通知
   │
6. 結果を audit_logs に記録
```

### 12.4 不当評価アラートの段階的エスカレーション

AI が「不当評価の可能性」を検出した場合、即座に人事へ通知するのではなく、段階的に対応する。

| 段階 | トリガー | アクション |
|------|----------|-----------|
| **Level 1: 観察** | 成果データと定性評価に 1 回の乖離を検出 | AI が内部フラグを立てる。外部への通知は行わない |
| **Level 2: 確認促進** | 3 回連続で乖離が検出 | 上司に対し「評価の根拠を具体的に記録してください」というリマインドを表示 |
| **Level 3: 人事レビュー依頼** | Level 2 のリマインド後も改善が見られない（2 ヶ月経過） | 人事担当者に「確認依頼」として通知（アラートではなく確認依頼として提示し、AI 判定の不確実性を明記する） |

### 12.5 AI 障害時のフォールバック

| 障害シナリオ | フォールバック動作 |
|-------------|-------------------|
| **音声書き起こし API の障害** | 手動メモ入力モードに自動切替。セッションは中断せず継続する |
| **AI 要約生成の障害** | 書き起こしテキストのみを保存し、要約は障害復旧後にバッチ生成する |
| **MQ スコア算出の障害** | 当月のスコア算出をスキップし、次月に 2 ヶ月分を合算して算出する。人事への通知は行わない |
| **判定レビューシステムの障害** | 新規アラートの発出を一時停止する。復旧後に未処理分をまとめて生成する |

| **データベース** | Amazon Aurora MySQL 8.0+ | MySQL 互換で高可用性。リードレプリカでダッシュボード系の読み取り負荷を分散可能 |
| **キャッシュ** | Amazon ElastiCache（Valkey / Redis 互換） | 既存の docker-compose に Valkey があるため親和性が高い |
| **音声ファイル保存** | Amazon S3（暗号化：SSE-S3 or SSE-KMS） | 30 日後の自動削除は S3 ライフサイクルポリシーで実現 |
| **音声書き起こし** | OpenAI Whisper API（第 1 候補）/ Amazon Transcribe（第 2 候補） | Whisper は日本語精度が高くコスト安。Amazon Transcribe は AWS 内完結でデータ主権に優れる |
| **話者分離** | Amazon Transcribe（話者分離内蔵）or pyannote.audio on ECS | Amazon Transcribe なら STT + 話者分離が 1 サービスで完結し構成がシンプル |
| **LLM 分析** | Amazon Bedrock（Claude） | AWS 内完結でデータが AWS 外に出ない |
| **非同期処理** | Amazon SQS + Laravel Queue | 音声処理パイプライン（STT → LLM 分析 → 結果保存）をジョブチェーンで実装 |
| **バッチ処理** | Laravel Schedule + Amazon ECS Task | MQ スコア月次算出等 |
| **認証** | AWS Cognito（SAML / OIDC 対応）or 既存の JWT 認証を拡張 | SSO / MFA 要件を満たす |
| **監視** | Amazon CloudWatch + Laravel Telescope（開発時） | — |
| **CI/CD** | GitHub Actions | 既存の `.github/workflows/main.yml` を拡張 |

## 15. AI 乱用防止設計

### 15.1 基本方針

本システムの AI 機能は「支援」を目的として提供する。AI の分析結果が人事上の懲罰・圧力・監視の道具として転用されることを、技術的・制度的の両面から防止する。

**原則:**

1. **AI は助言するが、決定しない** — AI の出力はすべて「参考情報」であり、人事判断の最終決定権は人間にある
2. **透明性の担保** — 利用者は「自分に関するどのデータが、誰に、どう使われているか」をいつでも確認できる
3. **抑止と検知の両立** — 乱用を事前に制限する仕組みと、事後に検知する仕組みを両方備える

### 15.2 想定される乱用パターンと対策

#### パターン 1: AI 分析結果の武器化（上司 → 部下）

上司が AI の分析結果（発話比率、感情スコア等）を部下への叱責・圧力の根拠に悪用するケース。

| 対策 | 実装方式 |
|------|----------|
| **AI 出力への注釈義務** | すべての AI 分析結果の表示画面に「この情報は対話改善の参考です。人事評価や懲罰の根拠としての使用は禁止されています」という注釈を常時表示する |
| **コピー・スクリーンショット抑止** | AI 分析画面のテキスト選択を無効化し、`PrintScreen` キー押下時に警告を表示する（完全防止は不可能だが、抑止効果を狙う） |
| **「引用禁止」ポリシー** | AI の分析結果を 1on1 中に部下に直接見せたり、評価面談で引用したりすることを利用規約で禁止する |
| **部下からの通報チャネル** | 「AI の分析結果を不当に使用された」と感じた場合の匿名通報フォームを設ける |

#### パターン 2: 過剰閲覧・ストーキング（上司 → 部下）

上司が特定部下のデータを業務上の必要性を超えて繰り返し閲覧するケース。

| 対策 | 実装方式 |
|------|----------|
| **閲覧回数の制限** | 上司が同一部下の詳細データ（書き起こし、感情スコア等）を閲覧できるのは、セッション後 7 日以内、最大 5 回までとする |
| **異常閲覧の自動検知** | 同一部下のデータを短時間に集中して閲覧した場合（例：1 時間に 10 回以上）、`audit_logs` を基に人事部門にアラートを送信する |
| **閲覧履歴の部下への開示** | 部下は「自分のデータを誰がいつ閲覧したか」を確認できる画面を提供する（透明性による抑止） |

```sql
-- 異常閲覧検知クエリの例
SELECT user_id, target_id, COUNT(*) AS view_count
FROM audit_logs
WHERE action = 'view_session_detail'
  AND created_at >= NOW() - INTERVAL 1 HOUR
GROUP BY user_id, target_id
HAVING COUNT(*) >= 10;
```

#### パターン 3: スコア操作（台本化・やらせ 1on1）

上司と部下が共謀して 1on1 を台本化し、AI スコアを意図的に高く見せるケース。

| 対策 | 実装方式 |
|------|----------|
| **パターン類似度検知** | 同一ペアの直近 3 回のセッションで、発話構造（話題の順序、発話比率、キーワード）が極端に類似している場合にフラグを立てる |
| **形式的セッション検知** | セッション時間が極端に短い（10 分未満）、トピックが毎回同一、アクションアイテムが毎回空の場合に形骸化アラートを発出する |
| **スコアの急激な改善への懐疑** | MQ スコアが 1 ヶ月で 20 ポイント以上上昇した場合、自動的にレビュー対象とする |

#### パターン 4: フィードバック攻撃（部下 → 上司）

部下が虚偽の低評価フィードバックを繰り返し、上司の MQ スコアを意図的に下げるケース。

| 対策 | 実装方式 |
|------|----------|
| **外れ値の統計的検出** | 同一上司に対する複数部下のフィードバックのうち、1 名だけが極端に低い評価を継続している場合にフラグを立てる |
| **フィードバックと対話内容の整合性チェック** | フィードバックで「不満足」と回答しているが、対話中の感情スコアや発話内容に不満の兆候がない場合、AI が不整合として記録する |
| **単独フィードバックの影響制限** | MQ スコアの算出において、特定の 1 名の部下からのフィードバックが全体に与える影響を最大 20 % に制限する（外れ値のトリミング） |
| **一定期間の冷却** | 同一上司への連続した低評価（3 回連続で最低評価）があった場合、4 回目以降は人事レビューを経なければスコアに反映しない |

#### パターン 5: 権限外のデータアクセス（人事・経営）

人事担当者が業務範囲外の社員データを閲覧する、または経営層がアクセス制御を迂回するケース。

| 対策 | 実装方式 |
|------|----------|
| **アクセス目的の入力必須化** | 人事担当者が `transcript_text`（書き起こし生テキスト）にアクセスする際、「確認目的」の入力（プルダウン選択 + 補足記述）を必須とする |
| **一括エクスポートの禁止** | 10 件以上のセッションデータを一括ダウンロードする機能を提供しない。個別セッション単位でのみ閲覧可能とする |
| **定期的なアクセスレビュー** | 四半期ごとに、人事・経営層のアクセスログをコンプライアンス部門がレビューし、不適切なアクセスがないか確認する |
| **Need-to-Know 原則** | 人事担当者であっても、自部署の担当範囲外の社員データには原則アクセスできない。部署横断のアクセスには上長承認を必要とする |

#### パターン 6: API・システムリソースの濫用

大量リクエスト、データ一括取得、自動化ツールによる不正利用。

| 対策 | 実装方式 |
|------|----------|
| **レートリミット** | ユーザーあたり 60 リクエスト/分、IP あたり 120 リクエスト/分を上限とする |
| **LLM API 利用上限** | ユーザーあたり月間 500 回の AI 分析リクエストを上限とし、超過時は管理者承認を要求する |
| **セッション作成上限** | 同一ペアの 1on1 は週 3 回まで。これを超える場合は「業務上の合理的理由」の入力を求める |
| **データエクスポート制限** | CSV / PDF エクスポートは 1 日 10 件まで。超過時はアラートを発出し、24 時間のクールダウンを設ける |
| **自動化検知** | User-Agent や操作間隔の分析により、Bot やスクレイピングツールによるアクセスを検知・遮断する |

#### パターン 7: 対話中の AI 分析誘導（プロンプトインジェクション的行為）

対話中に AI の分析結果を操作する意図で特定のキーワードや発言パターンを仕込むケース。

| 対策 | 実装方式 |
|------|----------|
| **単一指標への非依存** | AI の品質スコアは、キーワード出現頻度だけでなく、発話比率・話題の展開・事後フィードバック等の複数指標を組み合わせて算出する。特定の発言だけでスコアを大きく操作することが構造的に困難な設計とする |
| **テンプレート発話の検知** | 「傾聴しています」「あなたの意見を尊重します」等の定型句が不自然に高頻度で出現した場合、品質スコアへの寄与を減衰させる |
| **経時変化の重視** | 単発のセッションスコアではなく、3 ヶ月以上の傾向を重視する。1 回のセッションで極端に高いスコアを出しても、MQ スコアへの影響は限定的とする |

### 15.3 乱用検知のアラート体系

| レベル | 検知条件 | 通知先 | アクション |
|--------|---------|--------|-----------|
| **Info** | 閲覧回数上限の 80 % に到達 | 本人 | 注意喚起メッセージを表示 |
| **Warning** | 異常閲覧パターンを検知 / レートリミット超過 | 本人 + 管理者 | 一時的な機能制限（30 分）を適用 |
| **Critical** | 部下からの不正利用通報 / フィードバック攻撃の疑い / 一括エクスポート試行 | 人事 + コンプライアンス部門 | 当該ユーザーの機能を一時凍結し、調査を開始する |

### 15.4 利用規約への明記事項

以下を AXOL Loop の利用規約に明記し、全ユーザーに同意を求める。

1. AI の分析結果を人事評価・懲罰・配置決定の **唯一の根拠** として使用してはならない
2. AI の分析結果を 1on1 の場で部下に直接提示し、圧力をかける行為を禁止する
3. 他者のデータを業務上の正当な目的なく閲覧する行為を禁止する
4. 意図的にスコアを操作する行為（台本化、フィードバック攻撃等）を禁止する
5. 上記に違反した場合、システム利用停止および社内懲戒の対象となりうる

### 15.5 定期監査の実施

| 監査項目 | 頻度 | 実施者 |
|---------|------|--------|
| アクセスログの異常パターン確認 | 月次 | システム管理者 |
| 人事・経営層のアクセス妥当性レビュー | 四半期 | コンプライアンス部門 |
| 乱用通報の集計・傾向分析 | 四半期 | 人事部門 |
| 利用規約遵守状況の全体評価 | 年次 | 外部監査（推奨） |
| AI 分析精度の検証（乱用検知の誤検知率含む） | 半期 | 開発チーム + 人事部門 |
